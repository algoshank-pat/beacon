# Runbook — Getting Back to Work After a Restart

**Terminology note, added after real confusion this session**: "the scheduler"
means specifically `app.scheduler` (the persistent process below). A one-off
command like `python -m app.cli filter` or `ingest` run in a background
terminal is **not** the scheduler, even if it's running for a long time —
check `Get-CimInstance Win32_Process -Filter "Name='python.exe'"` and read the
actual command line before asking to "stop the scheduler."

The scheduler (`app/scheduler.py`) is built and runs as one long-lived
process, four independent scheduled jobs: main pipeline (seed-via-sheet →
ingest → filter → visa-scan → link-check → sort+resync) 3x/day at 8am/1pm/6pm Central; fit-scoring 3x/day
at 8:05am/1:05pm/6:05pm (5 minutes after the main pipeline, deliberately —
see "What changed recently" below); company enrichment also 3x/day, at
8:10am/1:10pm/6:10pm (10 minutes after the main pipeline); and the Approval
Poller every `approval_poll_interval_minutes` (default 30).
Fit-scoring used to run overnight-only (2am); it moved onto the same
daytime hours as the main pipeline once it became manually-gated by the
Beacon sheet's "My Decision" column rather than auto-scoring everything.
Company enrichment used to be fully manual-only, then once/day at 7pm, then
moved to this 3x/day cadence once cumulative daily-usage tracking made any
frequency safe; each firing now runs two independent passes -- StartupHub.ai
uncapped against the whole never-checked backlog (no published quota) and
FMP capped at the day's remaining `daily_enrichment_limit` (200/day, the
only source with a real quota) — see "What changed recently" below for all
of this. **Start
exactly one instance** — running two at once causes duplicate pipeline runs
and doubled Sheets API load (hit live more than once):

```powershell
.venv\Scripts\python.exe -m app.scheduler
```

**To keep it running automatically across restarts/logins** (the rest of M9),
run `scripts\register-scheduler-task.ps1` yourself from a normal PowerShell
window — Claude Code doesn't have permission to register scheduled tasks in
this environment (`Register-ScheduledTask` returned "Access is denied" when
attempted from the session). It sets up a per-user "at logon" trigger with
restart-on-failure (999 retries, 1-min interval) via
`Register-ScheduledTask`/`New-ScheduledTaskSettingsSet` — no admin rights
needed for a logon trigger:

```powershell
powershell -File C:\AI\beacon\scripts\register-scheduler-task.ps1
```

Until you run that once, if the process dies or the machine restarts, start
it again manually with the command above. Everything below still works as
one-off manual commands too, scheduler running or not.

**Known issue**: running the scheduler as a background job tied to a Claude
Code session (rather than via the script above) has died unexpectedly
several times, always right as the pipeline's `ingest` step starts, with no
Python traceback — `ingest` runs fine standalone (`python -m app.cli ingest`,
confirmed live), so this looks like the hosting session's background-process
lifetime, not an application bug. The Scheduled Task above runs independently
of any chat session and doesn't have this problem.

**Separate, more serious issue, now guarded against**: a single scheduler
launch (from a Claude Code session's Bash tool, and separately from a normal
user PowerShell window) was observed more than once to result in *two*
independent scheduler processes running concurrently against the same DB and
live Sheet — confirmed via overlapping `workflow_runs` rows, not just two
PIDs in the process list. Root cause was never pinned down. `app/scheduler.py`
now takes an exclusive lock on `scheduler.lock` at startup and refuses to run
if another instance already holds it, logging a clear error instead of
silently duplicating writes. **Still worth a habit**: before starting the
scheduler, or if something seems off (duplicate-looking Sheet writes, jobs
processed twice), check `Get-CimInstance Win32_Process -Filter "Name='python.exe'"`
and kill anything unexpected before restarting.

## What changed recently

**Already built:**
- **Company enrichment moved from fully-manual to scheduled** (per direct request, "run it on schedule keeping api quota in mind"): a new `run_scheduled_enrichment()` job runs, capped at a new `daily_enrichment_limit` filter_setting (default 100 companies/day). Sized around **FMP's confirmed 250-requests/day free-tier limit** (checked live against FMP's published pricing page — not guessed) — each company costs 1 FMP call (private, no ticker match, the common case) or 2 (publicly traded: search + profile), so 100/day stays under budget even at the 2-call worst case. StartupHub.ai has no published rate limit, so the cap is sized around FMP's harder, confirmed constraint instead. The manual `enrich-companies` CLI command still exists for on-demand runs. **Scheduler "optimized" alongside this**: fit-scoring moved from firing at the *exact same instant* as the main pipeline (8/1/6 o'clock sharp) to 5 minutes later (8:05/1:05/6:05) — APScheduler's thread-pool executor can genuinely run same-instant cron jobs concurrently, and this was a real contributor to the 429 quota errors hit repeatedly this session; enrichment runs 10 minutes later still (8:10/1:10/6:10) for the same reason. 571 companies with a job currently on Beacon had never been enriched at the time this was found — expect it to take about 6 days to drain that backlog at 100/day before settling into steady-state (new companies only).
- **Enrichment moved from once/day to 3x/day** (per direct follow-up request, "Can we invoke enrichment more than once a day?"): `run_scheduled_enrichment()` was rewritten to track *cumulative* usage across the whole day rather than a flat per-run cap — a new `get_enriched_today_count()` helper in `app/enrichment.py` counts companies whose `financial_data_last_checked` falls today (same pattern `app.budget.get_spend_today` uses for the LLM token budget), and each run only claims `daily_enrichment_limit` minus whatever's already been used today. This is what makes firing the job 3x/day safe — each invocation just picks up wherever the daily budget left off, so it can't blow past `daily_enrichment_limit` no matter how many times it fires. The `enrich_companies` cron changed from once/day at 7pm to `hour="8,13,18", minute=10` (10 minutes after main_pipeline, alongside fit-scoring's 5-minute offset), matching the main pipeline's cadence. A run that finds the daily cap already reached returns `{"skipped": True, "reason": "daily_enrichment_limit already reached today"}` and does nothing.
- **`daily_enrichment_limit` raised from 100 to 200/day** (per direct follow-up, "would I exceed quota... any enrichment which is free and has no quota, run it more frequently"): live data at the time showed 0 of the first 7 enriched companies were public matches (the 2-FMP-call case), so real-world cost is running close to 1 call/company, not the assumed worst-case 2 — meaning the old 100/day cap only used ~40% of FMP's real 250-requests/day quota even in the worst case. 200/day keeps worst-case usage (400 calls, only if every company were public — essentially never happens for this startup-heavy company list) still bounded by the cumulative-tracking mechanism, and realistic usage (~200-220/day) stays under the 250 ceiling.
- **Enrichment split into two independent passes** (per direct follow-up, "try searching for every company using startuphub.ai and then leave the rest to fmp... since it's free, we can just hit this API more times"): reordering alone was a dead end — FMP is the *only* source for employee_count/company_type/funding_stage/revenue_or_valuation, so it has to be queried for every company regardless of call order, meaning order never reduced FMP's call volume. The actual fix was decoupling the two sources' "already checked" tracking: migration `0004_add_startuphub_last_checked.sql` adds a `startuphub_last_checked` column alongside the existing `financial_data_last_checked`. `app/enrichment.py` now exposes `run_startuphub_enrichment()` (uncapped, runs against the entire never-checked-by-StartupHub backlog every time — no quota reason to throttle it) and `run_fmp_enrichment()` (capped at the day's remaining `daily_enrichment_limit`, since FMP is the one with a real quota) as two separate functions; `run_enrichment()` still exists as a combined wrapper for the manual `enrich-companies` CLI command. `run_scheduled_enrichment()` in `app/pipeline.py` calls both directly with their respective caps. A real, immediately-visible side effect confirmed live: Booz Allen Hamilton, already FMP-checked earlier this session (a confirmed public match), got `founded_year=1917` filled in by a StartupHub-only re-check that the old design's "skip StartupHub on an FMP match" rule would never have allowed — FMP's own profile endpoint never exposes `founded_year` at all, so public companies were permanently missing that field before this change. 576 companies are eligible for the (uncapped) StartupHub backlog as of this change; the next scheduled enrichment run processes all of them in one pass.
- **Live crash + a real, pre-existing data-quality bug found while draining the new StartupHub backlog**: the first uncapped StartupHub run crashed mid-batch on a `gspread.exceptions.APIError: [503]` from `find_existing_row` — `app/sheets_retry.py`'s `call_with_retry` only retried HTTP 429, not Google's other transient 5xx codes, so a single hiccup partway through a 576-company run took the whole batch down. Fixed by widening it to the same `RETRYABLE_STATUS_CODES` set (`{429, 500, 502, 503, 504}`) `app/http_client.py` already uses for every other API in this pipeline, and by wrapping each company's `_push_to_beacon` call in its own try/except in both `run_startuphub_enrichment`/`run_fmp_enrichment`, so a persistent Sheets outage on one company logs and moves on instead of aborting the rest of the backlog. Re-running the full backlog after this fix then surfaced a second, unrelated, **pre-existing** bug: `app/fmp.py`'s `search_symbol()` had no name-verification step at all — it just trusted FMP's fuzzy name search's first result. With only 7 companies ever checked before this session, this was invisible; running FMP against ~250 companies in one day exposed it at scale — confirmed live, "Saic" (real ticker SAIC) matched Mosaic's ticker instead, "Bloomberg" matched an unrelated gold-mining stock, "Google" matched an unrelated micro-cap, plus ~12 more (RSM, SHI, LMI, ECS, MPD, IDEX, Sage, Sentinel, Healthie, AeroTEC, "Executive Search", "Technology Company" — the latter two are clearly bad/placeholder names in the seed list to begin with). Fixed by adding `_names_plausibly_match()` — requires every significant word of whichever name is shorter to appear as a whole word in the other, after stripping common corporate suffixes (Inc/Corp/LLC/etc) — before `search_symbol` trusts any candidate. Deliberately conservative (not acronym-aware — "Saic" no longer matches "Science Applications International Corp" either, since acronym support would risk reintroducing exactly this bug for other short 2-4 letter names like RSM/LMI/ECS/MPD/SHI): in keeping with this pipeline's established "leave it blank rather than risk being wrong" enrichment policy, a real match lost to over-caution is a much smaller problem than another wrong one slipping through. The ~15 companies corrupted by the live crash-before-the-fix were fully reset (every enrichment field plus both `financial_data_last_checked`/`startuphub_last_checked` nulled, confirmed neither still held any legitimate StartupHub data — StartupHub had run first for all of them and found nothing, so the wrong data was 100% FMP's) and their Beacon rows re-pushed blank; they'll get a clean, correctly-verified re-check on a future run.
- **Visa-scan full-page fetch wired in** (user reported seeing many descriptions with a "we typically do not offer visa sponsorship..." disclaimer that wasn't getting caught): root-caused, not guessed at — queried the live DB and found the *median* stored `description` length across all 2,380 jobs then on Beacon was exactly 500 characters (Adzuna's server-side truncation cap), and the 1,493 jobs classified `no_mention` (no sponsorship language found at all) also had a median length of exactly 500. Since sponsorship disclaimers are almost always EEO/legal boilerplate near the *end* of a real posting, truncation was cutting them off before the keyword pre-check ever saw them — the classification logic itself was working correctly on the (incomplete) text it had. Fixed by wiring `app.salary_extraction.fetch_job_page_text()` — an existing, tested, previously-uncalled helper originally built for salary extraction — into `run_visa_scan()`: every job now gets its full posting page fetched before classification, replacing the (possibly truncated) stored description; falls back silently to the stored description on any fetch failure (dead link, network error) so one bad URL can't block the rest of a scan run. Chosen deliberately as "fetch for every job" rather than a narrower "only when 500-char truncation is suspected" heuristic — simpler, and doesn't risk missing genuinely-short real postings. 176 jobs were queued for visa-scan when this shipped; expect the next run to take noticeably longer than usual (one extra HTTP request per job) before settling back to normal.
- **Keyword-matching false-positives fixed (three distinct root causes, found via real Beacon postings the user flagged as irrelevant)**:
  1. **"X12" shift-notation collision, two variants**: a nursing job ("RN - ICU") passed because `tech_keyword_include`'s "X12" (EDI standard) substring-matched inside "Weekx12" (fixed with a leading-word-boundary requirement); a second SSM Health posting ("Respiratory Care Practitioner") still passed via "1 to 2 x12-Hour Shifts/Week" (a genuine word boundary before "x12" here, since preceded by a space) — fixed by also excluding a match immediately followed by "-hour"/" hour". "X12" was kept, not removed — it has hundreds of genuine EDI matches (EDI Analyst, IBM Sterling B2B Architect, Software AG webMethods Consultant, etc.) in the DB. Neither fix touches the end-anchor already relied on for "Architect"→"Architecture"/"Engineer"→"Engineering" suffix matching.
  2. **Company's own name as a tech keyword**: Kong and Boomi are both tracked companies whose own name is *also* a `tech_keyword_include` entry (both are API-Gateway/iPaaS vendors named after their product) — every posting from them mentions their own name in ordinary boilerplate, so literally any role (Renewal Account Representative, Sales Development Representative, Product Marketing Manager) trivially "matched." Fixed in `evaluate_job` by excluding any keyword that case-insensitively equals the posting's own company name — the same keyword still counts normally for any *other* company's posting that genuinely requires that tool as a skill.
  3. **`require_us_location` missing several real cities**: "Dublin" and "Paris" (bare, no country name attached) passed as US locations — `app/location.py`'s `NON_US_SIGNALS` list simply didn't have them. Auditing further found ~15 more real gaps (Bengaluru, Tokyo, Zurich, Dubai, etc.) plus a whole unrecognized pattern: global companies (seen live: Stripe) encode office locations as ISO-3166 country-code prefixes ("AE-Dubai", "IN-Bengaluru", "JP-Tokyo") — added a structural check for that pattern too. Deliberately did NOT add bare "Manchester"/"Melbourne"/"Vancouver" as signals (real US namesakes: Manchester NH, Melbourne FL, Vancouver WA) — used specific compound forms instead where those cities needed to resolve as non-US.
  4. **Real bug in fix #3 itself, found one session later**: "South Paris, Oxford County" — a real Maine town — got misclassified as Paris, France, because bare "paris" was an unconditional non-US signal checked before any US signal. Real US namesake towns exist for several other city names added in fix #3 too (Dublin OH/GA, London OH, Berlin NH, Warsaw IN, Madrid NY, Lisbon OH). Restructured `_segment_is_us` into a two-tier check: unambiguous non-US names (no US namesake, e.g. "Herzliya") checked first, then a US confirmation (state abbreviation, state name, or a "County"/"Parish" suffix — Adzuna's real US locations always have one), and only *then* the "risky" city names with real US namesakes. Both directions verified live: "Herzliya, IL" still resolves as Israel (not Illinois) and "South Paris, Oxford County" now resolves as Maine.

- **"Cloud Platforms" column built** (per direct request: detect AWS/GCP/Azure mentions in title+description): new `app.cloud_platforms.resolve_cloud_platforms()`, computed once at ingest time same as `app.location_state`, appended to the end of both `MAIN_SHEET_COLUMNS`/`JOB_LOG_COLUMNS` (append-only, per the Location State column-move lesson from earlier in this project's history -- never insert mid-list on a live sheet). Comparing AWS vs. GCP vs. Azure mention counts surfaced that most jobs had no cloud label at all, which traced back to the familiar Adzuna 500-char description truncation (confirmed: 87.8% of Adzuna-sourced Beacon jobs truncated to exactly 500 chars, vs. 0% for Greenhouse/Lever/Ashby-sourced jobs) -- same root cause already fixed for visa-scan and salary. Built `app/cloud_platforms_refresh.py` (migration `0009_add_cloud_platforms_checked_at.sql`) as a refresh pass mirroring `app.salary_refresh`'s pattern exactly: re-checks via the full posting page for any job whose stored description is exactly 500 chars, same "some Adzuna redirects are unfetchable client-side JS" limitation as salary/link-check. New `cloud_platforms_refresh_batch_size` setting (200/run).
- **Major bug found and fixed: unbounded `ws.append_row()` calls were silently ballooning the Beacon sheet's column count, and this is what caused the mysterious 10M-cell workbook-limit errors that killed the `filter` pipeline step for 3+ days straight (2026-07-11 through 2026-07-14).** Root-caused the hard way: dimension checks kept showing column count growing (33 → 1965 → 2863 → ~2870, etc.) even with the scheduler fully stopped and only one script touching the sheet, ruling out concurrency, a stale scheduler process, and every explicit `resize`/`insert_cols` call in the codebase (there's exactly one, and it's rows-only). Isolated it via a controlled single-job trace: calling `add_job_to_beacon` for one real job, with `table_range` explicitly bound to the actual column width, held column count perfectly flat across 8 consecutive real appends; the exact same appends *without* that bound grew column count on nearly every single call (~1.5-2% per call -- consistent with Google's own grid-auto-expand heuristic kicking in when `values.append` is given an unbounded range, i.e. just the sheet name with no A1 bounds, to auto-detect "the table"). Fixed by adding an explicit `table_range=f"A1:{last_col}1"` to both `add_job_to_beacon` (`app/sheets.py`) and `upsert_job_log_row` (`app/job_log.py`)'s `append_row` calls -- the only two places in the codebase that ever append a row. All 396 tests pass (6 new: `test_sheets.py`/`test_job_log.py` fake-worksheet updates). **This bug had been silently blocking the entire filter step since 2026-07-11**, meaning a growing backlog of jobs never got filtered/added to Beacon or the Job Log during that window -- backlog-clearing is ongoing, throttled to avoid saturating the Sheets API while the sheet is in active use.
- **SmartRecruiters added as a 4th targeted ATS source** (per direct request, "our APIs are not effective... add more direct ATS sources"): new `app/sources/smartrecruiters.py`, wired into `SOURCE_FETCHERS`, `ingest_targeted_company`, and Seed-via-Sheet's probe order (Greenhouse → Lever → Ashby → SmartRecruiters). Structurally different from the other three: its list endpoint is paginated (100/page) and doesn't include the description or a human-facing posting URL -- both need a separate per-posting detail call, which would mean 500-1000+ extra HTTP calls per poll for a large board (Equinox alone has 666 open postings) if done for every posting on every cycle. Mitigated via an optional `known_urls` parameter -- only postings not already tracked get a fresh detail/description fetch; already-known ones just need their URL reconfirmed for closed-job detection. The posting URL doesn't need its name-derived slug (confirmed live: `https://jobs.smartrecruiters.com/{company}/{id}` with no slug resolves 200), so it's built directly from the paginated list without an extra call. Immediately found a real board on retry for 5 previously-"not found" companies: **Symphony AI** (`symphonyai`, 1 open role) -- Deepgram and "ode with Anthropic" separately matched on Ashby (spot-checked both, real boards, not false positives).
- **Seeded 224 companies extracted from a Notion-exported, image-only PDF** (`presales_opps1.pdf`) -- the PDF had zero extractable text (rendered/screenshot-style export), so pages were rendered to images (`pymupdf`, since the environment's built-in PDF reader needs `pdftoppm`, not installed) and manually transcribed page by page. Also seeded a follow-up batch (Ode, Symphony AI, Deepgram, "ode with Anthropic", Squaretalk, Graphwise, Inetum) per direct request -- see the SmartRecruiters entry above for how 3 of these resolved.

A 6th, genuinely different Lockheed posting was deliberately left unchanged since its real text was never confirmed.

**Still open / unresolved:**
- **`app.sheets_retry.call_with_retry` doesn't retry a raw network/DNS failure** (`requests.exceptions.ConnectionError`, e.g. `getaddrinfo failed`) -- only `gspread.exceptions.APIError` (429/5xx). Hit live during the post-fix backlog clear: a transient DNS blip crashed the run outright instead of backing off and retrying, unlike a quota/5xx error which already retries automatically. Not yet fixed.
- **Root cause of why the sheet's column count balloons in the first place is fixed** (see the `table_range` bug above), but keep an eye out for recurrence -- if `ws.col_count` on either live sheet is ever unexpectedly large again, check for another unbounded `append_row`/`append_rows` call before assuming it's external.
- **`app.fmp.search_symbol`'s name-verification can't distinguish two real companies sharing the same name**: confirmed live -- "Kong" matched an unrelated Beijing utility company (`0295.HK`) since that company's own registered name genuinely contains the word "Kong" too. No fix planned; would require a headless-browser renderer, a much bigger investment than this pipeline's other network calls, for a use case (salary text specifically) where the Adzuna-estimate fallback already covers the gap reasonably.

## 3. Run commands

The virtual environment is already built at `.venv` — you don't need to
"activate" it. Just call the venv's Python directly:

**PowerShell / cmd:**
```powershell
.venv\Scripts\python.exe -m app.cli <command>
```

**Git Bash:**
```bash
.venv/Scripts/python.exe -m app.cli <command>
```

(If you'd rather activate the venv so you can just type `python`, PowerShell:
`.venv\Scripts\Activate.ps1`, cmd: `.venv\Scripts\activate.bat` — then every
`python` call in that terminal window uses the project's venv until you close it.)

## 4. Available commands (as of this build)

| Command | What it does |
|---|---|
| `migrate` | Apply any pending DB schema migrations |
| `seed-companies --file seed_companies.yaml` | Load/update the company seed list |
| `seed-filters --file seed_filters.yaml` | One-time import of filter criteria (safe to re-run) |
| `seed-via-sheet` | Onboards companies flagged via Job ID=`SEED` rows on Beacon (guesses/verifies a Greenhouse/Lever/Ashby/SmartRecruiters board), and deletes rows already processed on a prior run. Free — no API credits needed |
| `ingest` | Poll Adzuna + Greenhouse/Lever/Ashby/SmartRecruiters for new postings |
| `filter` | Run the Filter Engine over jobs at `status='new'` — passing jobs land on Beacon immediately |
| `visa-scan [--limit N]` | Regex + free keyword pre-check + Haiku fallback visa-sponsorship classification; updates/evicts the Beacon row |
| `fit-score [--limit N]` | Scores only jobs flagged "Go Score" (or retries ones stuck "AI Score Pending") via the Beacon sheet's My Decision column — never the whole backlog. Also evicts any row flagged "Reject". Costs money (Sonnet 5), budget-capped |
| `enrich-companies [--limit N]` | Research company employee count/HQ/funding/etc. from FMP/StartupHub.ai only — always free; a field stays blank rather than ever spending on an LLM fallback. Runs both sources' passes back-to-back with `--limit` applied independently to each. Also runs on its own schedule now (3x/day at 8:10am/1:10pm/6:10pm — StartupHub uncapped, FMP's cumulative usage capped at `daily_enrichment_limit`, 200/day) — this command is for on-demand runs outside that schedule |
| `pipeline` | Runs `seed-via-sheet` → `ingest` → `filter` → `visa-scan` in one call — what the scheduler runs 3x/day |
| `approval-poll` | Reads the Decision column on every Beacon job awaiting one — Approve/Deny/stalled-reminder. What the scheduler runs every 30 min. **Careful running this manually against a large untouched backlog** — every stale "Pending" job past `approval_reminder_hours` gets a reminder write (and a real notification email) the moment you run it |

Note: both spec docs describe a `last-run` CLI command for checking the most
recent run's funnel and surfacing CRITICAL failures — it was never actually
built. For now, check `workflow_runs`/`step_logs` directly in `job_search.db`,
or `scheduler.log`.

There's no `notify` command anymore — a job gets its Beacon row the moment it
passes `filter`; `visa-scan`/`fit-score` update or evict that row in place.

**Example — process a small batch cheaply:**
```powershell
.venv\Scripts\python.exe -m app.cli visa-scan --limit 25
.venv\Scripts\python.exe -m app.cli fit-score --limit 25
```

## 5. Where things live

- **Database**: `job_search.db` (SQLite, in the project root)
- **Config/secrets**: `.env` (API keys, Sheet IDs — never committed to git)
- **Google service account key**: `service_account.json`
- **Your resume**: `resumes/base_resume.docx`
- **Main tracking Sheet**: "Beacon" (your matches — Decision column is where you Approve/Deny)
- **Job Log Sheet**: "Beacon Filtered" (every evaluated job, including filtered-out ones — no email notifications on this one, by design)
- **Tests**: `.venv\Scripts\python.exe -m pytest -q` (from the project root)

## 6. Budget check

Fit-scoring, visa-scan's Haiku fallback, and (for now, until the free-only
redesign lands) company enrichment cost real money and are capped by
`daily_token_budget` / `monthly_token_budget` (in `filter_settings`, default
$1.00/day, $15.00/month) — all three draw from the same shared pool. If a
command reports "budget exceeded," that's working as intended — it'll pick
back up once the budget resets (or you can raise it temporarily by updating
the `filter_settings` table directly). Once the enrichment redesign ships,
`enrich-companies` drops out of this shared pool entirely — it'll never call
an LLM, so it can't spend anything or compete with visa-scan/fit-scoring for
budget.
