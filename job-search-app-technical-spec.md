# Job Search Automation App — Technical Specification

## 1. Purpose
A personal, single-user workflow application that discovers new job postings, filters them, screens for visa sponsorship language, scores fit via the Claude API, and surfaces matches by writing to a Google Sheet — **notification is entirely native to Sheets** (Tools → Notification rules), not app-built. Approval happens via the Sheet's Decision column, polled on a schedule. On Approve, the app hands off resume/cover letter generation to an existing Claude Desktop project. Application/interview tracking lives entirely in the Sheet. **No Telegram, no bot, no webhook** — the one exception is a minimal direct-email fallback used only when the Sheets API itself is unreachable.

**Non-negotiable boundary**: the app never submits an application. It stops at generated materials + Sheet updates; submission is always manual.

**Reading this diagram**: the biggest structural change from the original design is that jobs no longer wait at the Fit Scorer gate to get a Sheet row — they get one the moment they clear the Filter Engine, and every later stage (visa-scan, fit-score, or a manual "Reject") either updates that row in place or evicts it to the Job Log. Fit-scoring is on its own schedule, same three daily times as the main pipeline, rather than chained after visa-scan in the same run — see §4. Seed-via-Sheet runs as the main pipeline's first step, ahead of Ingestion, so a company it onboards gets its postings pulled in during the same run.

## 3. Components

| Component | Responsibility | Tech |
|---|---|---|
| **Process Supervisor** | Keeps one long-lived Python process (`app/scheduler.py`) alive | Windows Task Scheduler, "at startup" + restart-on-failure — registration script (`scripts/register-scheduler-task.ps1`) written but must be run manually; Claude Code lacks permission to register scheduled tasks in this environment |
| **Scheduler** | `BlockingScheduler` with four jobs: main pipeline 3x/day via `CronTrigger` (8/1/6), fit-scoring 5 minutes after each of those same times via its own separate `CronTrigger` (8:05/1:05/6:05 — deliberately offset, not the same instant, since APScheduler's thread-pool executor can run same-instant cron jobs concurrently and this was a real contributor to live Sheets-API 429s), company enrichment 3x/day, 10 minutes after each of the same times via its own `CronTrigger` (8:10/1:10/6:10 — same offset reasoning), each firing running two independent passes (StartupHub.ai uncapped against the whole backlog, no published quota; FMP capped by cumulative daily usage, its only real quota — see below), Approval Poller via `IntervalTrigger` — all built — plus a startup catch-up run. Acquires an OS-level exclusive lock (`scheduler.lock`, `msvcrt.locking`) at startup and refuses to run if another instance already holds it — added after repeated live incidents of two independent scheduler processes running concurrently against the same DB/Sheet, root cause never conclusively identified | Python, APScheduler 3.11.3 |
| **Seed-via-Sheet** | Onboards a company from nothing but a name typed into Beacon's Company column on a row with Job ID = `SEED`. Guesses a handful of plausible board slugs (strip corporate suffixes, squash/hyphenate) and probes Greenhouse → Lever → Ashby → SmartRecruiters with each; accepts a match only if the guessed slug actually appears in the returned jobs' own URLs. On success, creates the company at `priority_tier='A'` and writes the outcome into the Title cell; on no match, writes a failure message instead. A row whose Title is already non-blank (outcome already seen) is deleted on the following run. Runs first in the scheduled pipeline, ahead of Ingestion, so a newly-onboarded company's postings flow in the same run | Python, `requests` (reuses `app.sources.*`) + `app.sheets` |
| **Ingestion Layer** | Broad (Adzuna — one query per keyword across two OR-pooled categories, `where`/`max_days_old` from filter criteria, also captures `adzuna_salary_min`/`max` as a permanent estimate) + targeted (Greenhouse/Lever/Ashby/SmartRecruiters, no query-time filtering) polling, dedup, retry/backoff, closed-job detection (now also evicts from Beacon, not just a DB status flip -- see below), fuzzy duplicate flagging. **Adzuna's closure gap covered separately** by `app.link_check` (re-fetches each job's own URL directly, since Adzuna's API can't be re-polled for a specific employer's current listings the way Greenhouse/Lever/Ashby/SmartRecruiters can) | Python, `requests` |
| **Filter Engine** | Live criteria: keyword match against title+description across two dynamic categories (`role_keyword_include`, `tech_keyword_include`), plus seniority/location/posted-date/company attributes. **A passing job gets a Beacon row immediately** (`add_job_to_beacon`) — this no longer waits on visa-scan or fit-score. Keyword matching excludes any keyword that equals the posting's own company name (fixes Kong/Boomi-style self-referential false positives — see §11), and `title_exclude` has grown to ~50 terms fielding real false-positive reports (industry-agnostic titles, company boilerplate, generic-English-word tech keywords at large/organizationally-diverse companies like OpenAI's 719 open roles) | Pure Python + `app.sheets` |
| **Settings UI** | Add/remove filter keywords, edit thresholds | FastAPI + Jinja2 HTML (localhost) — not built |
| **Visa Scanner** | Regex first, then a free keyword pre-check (`visa`/`sponsor`/`citizen`/`h1b`/`work permit`/`visa permit`/`work sponsor`/`visa transfer`) before ever calling Haiku — no keyword present anywhere → `No mention`, free, no Haiku call. A keyword present but regex inconclusive → `Visa Check Pending` (set before the Haiku call, in both the DB and on the Sheet; stays there and auto-retries next run if the call fails, without aborting the rest of that run's batch) → Haiku resolves it. Cuts Haiku's earlier ~99.7%-of-postings load down to just the smaller ambiguous-but-mentioned bucket. Never reads real DOL/USCIS sponsorship-history data — only the posting's own text (a distinct, unbuilt future item). Updates the job's existing Beacon row's Visa Flag in place (translated to a display label — `Sponsored`/`No sponsor`/`Unclear`/`No mention`/`Visa Check Pending`); evicts (deletes the row, moves it to the Job Log) if `restricted` + `require_visa_sponsorship` | Python `re` + Anthropic API + `app.sheets` |
| **Fit Scorer** | JD-vs-resume scoring, gated by the Beacon sheet's "My Decision" column (`Go Score` freshly requested, or `AI Score Pending` retried after a prior failed attempt — claimed before each API call so a failure leaves it retryable rather than stuck or silently reverted). Runs at the same 8am/1pm/6pm times as the main pipeline, as its own separate job/`workflow_run` — no longer needs off-peak isolation now that volume is manually bounded rather than the whole backlog. Also evicts any row with My Decision = `Reject` every run, independent of scoring. Updates the job's existing Beacon row's score in place (and flips My Decision to `AI Scored`) if above threshold; evicts to the Job Log if at/below | Anthropic API (Sonnet 5) + `app.sheets` |
| **Sheet Writer** (`app/sheets.py`) | Beacon: adds a row at filter-pass time, updates Visa Flag/Score/My Decision/Salary Range cells in place, evicts (deletes) rows, conditional-formats Decision-driven row color (Approve=green/Deny=red), score-highlight, Industry deprioritization grey, and Salary-Range-estimate pink; separately, a **static** (non-formula) Date-Posted-driven row color (today=light blue, this calendar week=light green) recomputed in Python every pipeline cycle rather than via a live conditional-format rule -- the live-formula version crashed the Sheets mobile app on scroll — **this is also the entire notification mechanism**, since every write triggers Google's own native notification rule | `gspread` + service account |
| **Job Log Writer** (`app/job_log.py`) | Upserts (find-by-Job-ID, create-or-update — not append) one row per excluded job to a **separate** spreadsheet with notifications deliberately disabled: filtered-out (with reason), visa-restricted-evicted, scored-below-threshold-evicted, or user-rejected (`My Decision = Reject`). Decision column (Accept/Reject) with the same row-color pattern as Beacon; never overwritten by automated updates | `gspread` + same service account, different spreadsheet ID |
| **Sheets Retry** (`app/sheets_retry.py`) | Wraps every Beacon/Job Log Sheets call with retry-and-backoff on 429 (`gspread.exceptions.APIError`) — hit live during the first bulk backlog run, which logs hundreds of jobs in a tight loop | Python |
| **Approval Poller** | Internal schedule (default every 30 min) reading the Decision column; Approve (records decision only, see Resume/Cover Letter Handoff)/Deny/reminder logic; idempotent via `decision_processed_at`; logs consecutive Sheets-API failures as CRITICAL entries to `step_logs` (`count_consecutive_failures`) — `last-run` CLI referenced as the surfacing mechanism doesn't exist, check `workflow_runs`/`step_logs` directly for now | Python, APScheduler job within the main process — **built (M6)**, `app/approval.py`; not yet exercised end-to-end against the live Sheet |
| **Resume/Cover Letter Handoff** | Composes prompt, clipboard copy, triggers Claude Desktop; also writes the prompt to a Sheet column as a fallback/confirmation | Python (`pyperclip`, `os.startfile()`) — not built (M7); the Approval Poller's Approve path currently stops at recording the decision |
| **`save-resume`/`save-cover-letter` CLI** | Registers manually-generated files, updates DB + Sheet | Python CLI — not built (M7) |
| **Company Enrichment** | **Always free.** The earlier Sonnet 5 + web-search fallback tier (~$0.15–0.75/company, the most expensive per-unit cost anywhere in this pipeline) was removed entirely — two free sources only: (1) Financial Modeling Prep — public companies only, complete data; (2) StartupHub.ai — AI-startup directory, partial data (HQ/founded/industry only on the free tier). If neither has a field, it stays blank for manual fill-in; no LLM call regardless of budget headroom (`enrich-companies` no longer even requires `ANTHROPIC_API_KEY`). Known consequence: a meaningful fraction of companies (private ones StartupHub doesn't cover) will simply stay unenriched. Scoped to companies with a job **currently on Beacon** (`jobs.sheet_row_number IS NOT NULL`) — changed from an earlier `status='notified'` check once jobs stopped being gated behind a score. Pushes results onto every already-existing Beacon row for that company (`app.sheets.update_company_columns`, called from every success path) — a real gap fixed earlier, since a job typically gets its Beacon row before its company is ever enriched. **Two independent scheduled passes** (`run_startuphub_enrichment`/`run_fmp_enrichment`, 3x/day at 8:10am/1:10pm/6:10pm) as well as a combined `run_enrichment` wrapper for manual/CLI use — StartupHub uncapped (no published quota, runs the entire never-checked backlog every time), FMP capped at `daily_enrichment_limit` (200/day, tracked *cumulatively* across all of a day's runs via `get_fmp_enriched_today_count()`, not a flat per-run cap) to respect its confirmed 250-requests/day free-tier limit. The two sources track "already checked" independently (`financial_data_last_checked` for FMP, `startuphub_last_checked` for StartupHub, migration `0004`) so a company already FMP-matched can still be revisited by StartupHub for fields FMP's own profile endpoint never exposes (`founded_year`, notably) | Python, `requests` (FMP/StartupHub) |
| **Salary Extraction** | Real range: regex over stored description, cheap-only at Beacon-add time (no page-fetch fallback there — runs for every passing job). `resolve_posted_salary()` (with the page-fetch fallback) is now wired in too, via a separate scheduled step, `app.salary_refresh` -- re-attempts extraction from the full posting page for any Beacon job still missing a real salary. Deliberately never conflates this with Adzuna's `salary_min`/`salary_max`, which are frequently their own algorithmic estimate — that estimate is instead captured permanently and separately as `jobs.adzuna_salary_min`/`max`, shown on Beacon in its own clearly-labeled columns as a fallback signal when the real range can't be extracted (the common case). **Real bug found and fixed**: `app/sources/adzuna.py` was writing Adzuna's estimate into `salary_min`/`salary_max` at ingest time, contradicting the design just described — fixed, and backfilled (16,487 affected DB rows nulled, ~1,158 corresponding live Beacon rows cleared via a new `update_salary_columns()`) | Python `re` + `requests` (with browser-like headers — some job boards 403 a bare request) |
| **Observability Layer** | `workflow_runs`/`step_logs`; `last-run` CLI **referenced throughout this spec but never actually built** — check `workflow_runs`/`step_logs` directly, or `scheduler.log` | Python, SQLite |

**Removed entirely from this design**: Telegram bot, inline buttons, approval webhook, and (post-M5) the standalone `notify` CLI command/`run_notify()` function — fit-scoring updates/evicts the Beacon row directly now, there's nothing left for a separate notify step to do. Notification is a Sheets-native feature you enable once in the UI, not application code.

## 4. Data Flow

**Main pipeline** (internal schedule, 3x/day at fixed times — default 8am/1pm/6pm, plus once immediately on process start; `app.pipeline.run_full_pipeline`):
1. Seed-via-Sheet onboards any Beacon row with Job ID = `SEED`: guesses/verifies a Greenhouse/Lever/Ashby/SmartRecruiters board from the typed company name, creates the company (`priority_tier='A'`) on a match, writes the outcome to the Title cell, and deletes rows already processed on a prior run (Title already non-blank)
2. Ingestion polls Adzuna — one query per active keyword across `role_keyword_include` + `tech_keyword_include` (using `what_phrase` for exact multi-word matching, which Adzuna checks against both title and description), `where` from `location_include` if set, `max_days_old` from `posted_within_days`; skipped with a System Log entry if no keywords are configured in either category — plus configured ATS boards (no query-time filtering possible, return all currently open postings, now including any company Seed-via-Sheet just added), dedupes, retries, flags closed/duplicate jobs — first run already captures roughly the last month by default. Adzuna results also carry a salary estimate, captured permanently into `jobs.adzuna_salary_min`/`max`
3. Filter engine evaluates against live criteria, including posted-date cutoff and `require_us_location` (a fail-open heuristic over free-text location strings, not a geocoder — see the build spec's Filter Criteria section). **A passing job gets a Beacon row immediately** (`add_job_to_beacon` — Job ID/Company/Title/link/location, description-only salary extraction, score and visa flag left blank); a filter-out is upserted to the Job Log with the specific reason — **except a non-US-location filter-out, which is silently dropped with no Job Log write at all** (`FilterResult.skip_log`), per direct request, since these are high-volume noise not worth logging. Seniority and remote-work type aren't sourced from any ingestion API; the Filter Engine infers both from title/location text on first evaluation
4. Visa scanner (regex → free keyword pre-check → Haiku fallback only for the remaining ambiguous bucket) updates the job's Beacon row's Visa Flag cell in place. If `restricted` and `require_visa_sponsorship`, the row is deleted from Beacon and upserted to the Job Log instead (`Visa Restricted` reason)

**Fit-scoring** (`app.pipeline.run_scheduled_fit_scoring`; its own separate schedule at the same 8am/1pm/6pm times as the main pipeline — used to run overnight at 2am, deliberately not chained after step 4 so it got a fresh, uncontested `daily_token_budget` instead of competing with the main pipeline's same-day spend; that isolation mattered less once scoring became manually-gated rather than automatic across the whole backlog):
5. First, independent of scoring: any Beacon row with My Decision = `Reject` is evicted to the Job Log (`Rejected (My Decision)` reason) — checked every run since it's the same column fit-scoring already reads
6. Fit scorer calls the API only for jobs flagged via the Beacon sheet's My Decision column (`Go Score`, or `AI Score Pending` if a prior run's API call failed and left it there for retry — claimed by setting `AI Score Pending` immediately before the call), checked against token budget guardrails (guardrail breach → System Log tab entry, native notification fires; whatever doesn't fit in one run's budget rolls over to the next). If the score is at/below `fit_score_threshold`, the row is deleted from Beacon and upserted to the Job Log (`Scored Below Threshold` reason, with the score, My Decision = `AI Scored`). If above, the Beacon row's Fit Score cell is updated in place and My Decision flips to `AI Scored` — **this update is itself the notification** (the row was already there; the cell edit triggers Google's native rule), no separate append/send step

**Approval poll** (internal schedule, default every 30 min, same process — **built, M6**; `app.approval.run_approval_poll` + `app.pipeline.run_scheduled_approval_poll`):
7. For jobs at `sheet_row_number IS NOT NULL` with `decision_processed_at IS NULL`, read the Decision cell — **not `status='notified'`**, which nothing sets anymore post-Beacon-immediate-add (see §6)
8. `Pending` within window → no action. Past `approval_reminder_hours` **and only if "Reminder Sent At" is still empty** → update "Reminder Sent At" (the edit itself notifies you) — fires once per job, not once per poll cycle, to avoid emailing every 30 minutes about the same stalled job. Baseline is `jobs.notified_at`, which existed in the schema from the start but was never actually set until this milestone — `add_job_to_beacon()` now sets it, falling back to `first_seen_at` for jobs added before the fix
9. `Deny` → read Rejection Reason, `status='rejected'`
10. `Approve` → **today, only** `status='approved'` + `decision_processed_at=now`, logged to `step_logs`. Steps 11-12 below (compose handoff prompt, clipboard copy, trigger `claude://`, write "Handoff Prompt" column) are still **M7, not built** — out of scope for the Approval Poller milestone, which was scoped to poll/reminder/decision-recording mechanics only
11. Once registered via CLI, re-score via API, update the Sheet — **not built (M7)**
12. If the poller can't reach the Sheets API for `approval_poller_max_consecutive_failures` consecutive cycles, log a CRITICAL entry to `step_logs` (`count_consecutive_failures` walks recent `approval_poll` workflow_runs). **No `last-run` CLI exists** to surface this despite being referenced as the mechanism throughout this spec — check `workflow_runs`/`step_logs` directly, or `scheduler.log`, for now

**A note on concurrency**: since CLI commands run manually and independently (and can overlap the scheduler process or each other — a real scenario hit live), every DB write is `commit()`-ed *before* any Sheets API call in the loop, never left open across one. SQLite's `busy_timeout` (30s) handles brief legitimate contention; it can't compensate for a transaction held open across a slow, retrying Sheets call, which is what actually caused a live "database is locked" failure before this was fixed.

**Closed-job detection**: re-poll of previously-notified jobs; 404/filled → `status='closed'`, Sheet Status written directly (the one automated exception to hand-maintained Status).

## 5. Database Schema (Pipeline State Only)
Full DDL in the build-spec document. **SQLite runs in WAL mode** (`PRAGMA journal_mode=WAL`) plus **`busy_timeout=30000`**, necessary because CLI commands run manually and independently and can legitimately overlap each other or the scheduler process — WAL alone only lets a reader coexist with a writer, `busy_timeout` is what lets a second writer wait its turn. The deeper requirement, also enforced throughout: every DB write commits *before* any Sheets API call in the same loop iteration — see §4's concurrency note. Summary: `companies` (categorization + enrichment: employee count, HQ, funding stage, revenue/valuation, H-1B history), `jobs` (`decision_processed_at`, `sheet_row_number` — now doubles as the "is this job currently on Beacon" signal, not just a polling-mechanism field; `adzuna_salary_min`/`max` added in migration `0003`; `notified_at` existed since `0001_initial.sql` but was never actually written until the M6 Approval Poller build — `add_job_to_beacon()` sets it now), `fit_scores`, `resume_feedback`, `filter_settings`/`filter_keywords`, `workflow_runs`/`step_logs`. No relational tracking tables — see §6. No schema migration was needed for the Industry Beacon column (§6) — it's a Sheet-only addition, not a DB column (already had `companies.industry` since `0001`).

## 6. Google Sheets Integration (Tracking + Approval + Notification)
Service account auth via `gspread`. **Notification is a Sheets-native feature, not application code**: enable Tools → Notification rules → "Any changes are made" → "Email right away" once, in the Sheets UI. Because the app writes as a service account (a distinct identity from you), every automated write triggers an email; your own manual edits (Decision, Status) do not notify you back, avoiding self-generated noise. **This behavioral assumption is verified explicitly during M5**, not taken on faith.

**Known limitation, accepted deliberately**: the native notification email is typically generic ("a change was made"), not a formatted summary of what changed — you'll click through to see details, every time. Traded away against building/maintaining a message-composing integration. Upgradeable later via a Google Apps Script `onChange` trigger if this proves too vague in practice, but not built preemptively.

**"System Log" tab**: a second tab in the same spreadsheet (`Timestamp`, `Alert Type`, `Detail`) for operational alerts not tied to a specific job row — source-poller failures, budget-guardrail pauses. Covered by the same notification rule automatically. **Not built yet.**

**Decision column** is the one genuinely two-way field, on both Beacon and the Job Log. Every other automated field is one-way (app → Sheet). Hand-maintained fields stay one-way the other direction, except the read-only Status check before sending a reminder (avoids re-nagging about a decision made outside the Decision column) and the one-time closed-job Status write. Approval Poller failures are logged to `step_logs` as CRITICAL entries — no separate alert channel beyond the `last-run` CLI. **Decision now also drives conditional-format row color** on both sheets: Beacon's `Approve`/`Deny` → green/red row; Job Log's `Accept`/`Reject` → green/red row. Implemented via raw `addConditionalFormatRule` requests (`ws.spreadsheet.batch_update`) since gspread has no dedicated convenience method for conditional formatting.

**Beacon gets a row the moment a job passes the Filter Engine — this is the biggest structural change since M5.** The original design gated the Sheet row behind `score > fit_score_threshold`; per direct request ("I don't want to wait for fit score before job gets added onto Beacon sheet"), that gate was removed. `add_job_to_beacon()` creates the row with score/visa flag blank; `update_visa_flag()`/`update_score()` fill them in later in place; `remove_main_row()` evicts the row (deletes it) if the job later turns out visa-restricted or scores at/below threshold, at which point `app.job_log.upsert_job_log_row()` gives it a home on the Job Log instead. The standalone `notify` command/`run_notify()` function that used to own Sheet-append is gone — this logic is now split across `add_job_to_beacon`/`update_visa_flag`/`update_score`, called from the Filter Engine, Visa Scanner, and Fit Scorer respectively.

**Salary is app-owned, not hand-maintained** (changed from earlier drafts, per direct request): parsed from the actual job posting text, never from Adzuna's `salary_min`/`salary_max` fields, which turned out to be their own algorithmic estimate rather than the employer's posted range in real testing. At Beacon-add time only the cheap description-text extraction runs (no page-fetch fallback — happens for every passing job). **Since most JDs don't state a real range explicitly, the real range is blank far more often than not** — Adzuna's estimate is shown as a fallback instead, in the single "Salary Range" column, suffixed `" (est.)"` and pink-highlighted (captured once at ingest into `jobs.adzuna_salary_min`/`max`, never overwritten, never conflated with a real range even when both exist — see `format_salary_range()`). **Real bug, found live and fixed**: `app/sources/adzuna.py` was writing Adzuna's estimate directly into `salary_min`/`salary_max` at ingest time, so any job whose real JD text didn't separately restate a salary (worsened by Adzuna's description being capped at 500 chars server-side — see the build spec's Job Source Ingestion section) permanently showed the estimate mislabeled as the real range. Fixed and backfilled: 16,487 DB rows corrected.

**Industry column, added post-M6 per direct request**: originally appended as Beacon's last column; now lives right after Company/Title in the redesigned 20-column layout below, since the redesign was a full rebuild anyway. Supports deprioritizing, not excluding, consulting/IT-staffing companies that were crowding out direct-employer postings — a conditional-format rule greys out the row when the Industry text matches a staffing/consulting/recruiting/professional-or-managed-services/outsourcing keyword, though a manual Decision or a populated Score always takes visual precedence if a row matches both.

**Full Beacon + Job Log column redesign — built and deployed to the live sheets** via a one-time `scripts/rebuild_sheets.py` (dry-run by default, `--confirm` to write; clears each sheet's cell values *and* strips its old conditional-format/data-validation rules before calling `ensure_main_sheet_headers`/`ensure_job_log_headers`, so new rules don't stack on top of stale ones). New Beacon layout (20 columns, `app.sheets.MAIN_SHEET_COLUMNS`): `Job ID, Company, Title, Industry, Location, Date Posted, Salary Range, Visa Flag, Decision, Rejection Reason, Reminder Sent At, My Decision, Initial Fit Score, Final Fit Score, Employee Count, Public/Private, Funding Stage, Revenue/Valuation, Apply URL, Application Status`. Salary Min/Max/Adzuna-Estimated-Min/Max (4 old columns) consolidated into one "Salary Range" column (`format_salary_range()`) with a light-pink cell background (keyed off a `" (est.)"` text suffix) as the Adzuna-estimate-fallback signal. Visa Flag's internal DB values are translated to display labels only at write time (`display_visa_flag()`): `Sponsored`/`No sponsor`/`No mention`/`Visa Check Pending`/`Unclear`. A new "My Decision" column (independent of the existing Decision approval-gate column) replaces the old Score Request checkbox with a fuller state machine: `New` (app default) → `Go Score` (user requests scoring) → `AI Score Pending` (app, claimed before each fit-scoring API call, stays here on failure for automatic retry) → `AI Scored` (app, done) / `Manual Scored` (user, independent of the AI path) / `Reject` (user — evicts the row to the Job Log, handled by `fit_scoring._process_rejections()` every fit-scoring run). A new fully-manual "Application Status" column (`Applied`/`Tech Rounds`/`Talent Acquisition`/`Panel`/`Rejected`) tracks post-application interview stages, fully independent of My Decision. ~12 previously-empty columns were dropped (confirmed empty on the live sheet first — Handoff Prompt, Recruiter info, Interview Notes, etc.). Job Log (19 columns, `app.job_log.JOB_LOG_COLUMNS`) mirrors Beacon's columns plus its own "Reason for Rejection" field, since the separate-spreadsheet architecture exists for notification isolation, not column-layout reasons. Both sheets got a clean rebuild rather than an in-place edit, since the reordering couldn't be done safely live; the live rebuild also repopulated Job Log from the **database** (767 excluded jobs, reconstructing visa-restricted reasons from `jobs.visa_snippet` and below-threshold reasons from `fit_scores`) rather than just the 182 rows already on the old sheet, since `job_log_enabled` had been off for a stretch and the sheet was missing most of the real exclusion history. Sheets-batching (bulk `append_rows()`, in-memory duplicate-check instead of a full-column read per job) is still **not built** for the live `filter`/`visa-scan`/`fit-score` steps — see "Still open" below; the rebuild script itself does batch its one-time writes.

**Job Log ("Filtered" sheet) — a second, separate spreadsheet, deliberately notification-free, redesigned multiple times since first built.** V1 logged every evaluated job (an append-only audit trail, including a `Passed Filter - Awaiting Score` row for every passing job — dropped from the rebuild as stale duplicates of what's already on Beacon). Once Beacon started getting a row immediately at filter-pass time, logging passing jobs to a second sheet too became redundant, so the Job Log's scope narrowed to **exclusions only**: filtered-out (never reached Beacon), visa-restricted (evicted from Beacon), scored-below-threshold (evicted from Beacon), or user-rejected (`My Decision = Reject`, evicted from Beacon). It's **upsert** (find-by-Job-ID, create-or-update), not append — one row per job, ever, matching how Beacon itself behaves, with a Decision column (Accept/Reject) the automated writers never overwrite. Same service account, `GOOGLE_JOB_LOG_SHEET_ID`, notifications never enabled — a queryable record you check on demand. Controlled by `job_log_enabled` in `filter_settings` (default `true`). This split (two spreadsheets, one high-volume/notification-free and one low-volume/notification-bearing) is a structural consequence of Beacon's notification mechanism — high-volume writes and notification-worthy writes can't share a spreadsheet under this design.

## 7. Human-in-the-Loop Boundaries
- **Never automated**: submitting applications, resume/cover letter generation itself, reading manually-edited Sheet cells back into pipeline logic (except Decision and the one-time closed-job Status write)
- **Always requires approval**: the deep-link handoff only triggers after the Approval Poller reads `Decision = Approve`
- **Never silent**: every pipeline step and poll cycle writes to `step_logs`; user-facing alerts land in the Sheet (triggering native notification) or, if the Sheet is unreachable, as CRITICAL entries in `step_logs` — the `last-run` CLI referenced here as the surfacing mechanism doesn't exist yet; check `workflow_runs`/`step_logs` directly, or `scheduler.log`, in the meantime
- **Never unattended past budget**: `daily_token_budget`/`monthly_token_budget` pause LLM-dependent steps and log to System Log
- **Pausable on demand**: `pipeline_paused` in `filter_settings` — **only the Approval Poller actually checks this today**; `run_full_pipeline`/`run_scheduled_fit_scoring` don't, so the pause switch doesn't yet stop the main pipeline or fit-scoring the way this line has always claimed. A real gap, not yet fixed. `--dry-run` on the CLI lets filter-criteria changes be validated against real data before they affect the live pipeline

## 8. Build & Deployment Model
- Development: Claude Code in auto-mode, inside a dev container
- **Runtime — native, not containerized**: required for the Claude Desktop handoff (`os.startfile()`/clipboard)
- **Single persistent process** (`app/scheduler.py`, `BlockingScheduler`): main pipeline (seed-via-sheet → ingest → filter → visa-scan, 3x/day at fixed times, default 8am/1pm/6pm via `CronTrigger` — matched to when postings actually cluster on weekday business hours, not hourly polling), fit-scoring (its own separate `CronTrigger`, same three daily times — used to run overnight at 2am, deliberately not chained after the main pipeline — see §4), and the Approval Poller (every `approval_poll_interval_minutes`, default 30, via `IntervalTrigger` — independent cadence since it's just a cheap Sheets read) as three internal APScheduler jobs inside one process, all built. One process avoids SQLite cross-process locking — enforced structurally, not just by convention, via an OS-level exclusive lock (`scheduler.lock`, `msvcrt.locking`) acquired at startup; a second launch refuses to run rather than silently duplicating. Added after repeated live incidents (confirmed via overlapping `workflow_runs`, not just a process-list artifact) of two scheduler processes running concurrently — root cause never conclusively identified, reproduced even from a single, deliberate, manually-typed launch. Windows Task Scheduler only keeps the process alive (at-startup trigger, restart-on-failure) — **this Task Scheduler wiring is the one remaining piece of M9**; the registration script (`scripts/register-scheduler-task.ps1`) is written but fails with "Access is denied" when run from within Claude Code, so it must be run by hand from a normal PowerShell window (not yet confirmed done); the persistent process itself is built and has been run live
- **Catch-up on sleep/crash/reboot**: the main pipeline job fires immediately on process start plus its regular schedule; generous `misfire_grace_time` (6 hours) on both jobs so sleep-missed runs fire when the process next gets a chance rather than being skipped. Fit-scoring catches up naturally over consecutive nights regardless (unscored jobs simply remain eligible)
- Secrets: `.env` + service account JSON, plaintext acceptable given this is a single-user personal machine

## 9. Cost Model — Token/API Cost Analysis
Google Sheets API calls (including the higher-frequency Approval Poller reads) remain free at this volume. Removing Telegram doesn't change cost (it was already free) but does remove one credential and one integration surface.

**Realized costs from live testing (M1–M5 + company enrichment), superseding the earlier estimate:**
- **Visa scan**: historically, regex confident on ~1/300 real postings, Haiku carrying almost all classification at ~$0.0007/job (637 avg input / 17 avg output tokens per call, measured from this account's real history) — clearing an 11,749-job backlog would have cost ~$8.45 at this rate. **Built**: a free keyword pre-check ahead of Haiku (visa/sponsor/citizen/h1b/work permit/visa permit/work sponsor/visa transfer) now resolves most postings that mention nothing about sponsorship at all to `No mention` for free instead of costing a Haiku call
- **Fit scoring**: ~$0.01–0.02/job on Sonnet 5 (no temperature control — see Fit Scoring Logic in the build spec — so this varies run to run for the same job). Volume is bounded by manual "My Decision = Go Score" flags rather than the whole backlog, so total spend here is driven by how many jobs the user actually asks about, not how many exist
- **Company enrichment**: **built — always $0.** The Sonnet+web-search fallback (previously ~$0.15–0.75/company, several times a single fit-scoring call) was removed entirely; only FMP + StartupHub.ai remain, both free
- **Seed-via-Sheet**: always $0 — probes Greenhouse/Lever/Ashby/SmartRecruiters's public job-board APIs directly, no LLM involved
- Adzuna, FMP, and StartupHub.ai calls are all free at this volume (free tiers)

**Realistic ongoing cost with the current design**: visa-scan and fit-scoring are the only things that can cost anything, and both are now bounded by real signals (keyword presence, manual flags) rather than raw posting volume — likely well under the old "$3-5/month" estimate, though not yet measured over a full month post-redesign. **Important shared-pool note, worth remembering**: visa-scan and fit-scoring draw from the same `daily_token_budget`/`monthly_token_budget` pool — company enrichment and Seed-via-Sheet are both outside this pool entirely, since neither ever calls the Anthropic API. Enrichment has its own, separate constraint instead: `daily_enrichment_limit` (200 companies/day, tracked cumulatively across 3x/day runs) sized around FMP's confirmed 250-requests/day free-tier cap, not a dollar budget.

**Backlog spikes from the company seed list growing from 6 to 154, in two batches**: the first `ingest` run pulled in 4,643 new jobs, the second (after the 89-company H-1B-sponsor-list batch) pulled in another 6,538 — pushing the `filter` backlog past 12,000 jobs at points, and separately leaving most companies then-current on Beacon never-enriched (moot for cost purposes now that enrichment is free-only, though many will simply stay unenriched since StartupHub doesn't cover most private companies).

**Sheets write-quota, not LLM cost, is the real bottleneck for `filter` at this scale** — every job evaluated costs ~2-3 Sheets API calls (a full-column duplicate-check read, plus an append or Job Log upsert) regardless of whether it passes. The original ~1,300-job backlog took 40+ minutes; the 12,000+-job backlog this session was going to take **4-6+ hours** at the same rate, which is why the Sheets-batching fix (bulk `append_rows()`, in-memory duplicate-check) was decided — see §6. `job_log_enabled=false` is an existing, no-code-change lever that cuts this roughly in half by skipping Job Log writes for filtered-out jobs entirely.

**Why filtering isn't done at ingest time to reduce this volume** (a real question raised and settled this session): keeping every ingested job in the DB even when it fails current criteria preserves the ability to retroactively re-evaluate against changed criteria later (sources don't reliably return the same posting again once it ages out), preserves debugging visibility ("did the pipeline even see this posting"), and keeps cross-source fuzzy dedup working (needs both candidates present). SQLite doesn't care about row count — the cost driver is Sheets writes, addressed directly above, not by moving filtering earlier.

**No real steady-state ingest rate is known yet** — every `ingest` run so far has coincided with either a company-list expansion or a process being killed/restarted mid-run, so there's no "just checking for new postings" data point to measure against. Worth revisiting via `jobs.first_seen_at` once the system runs stably for a week or two without the company list changing.

## 10. Resume Generation — Design Note
Unchanged: Claude Desktop's `claude://claude.ai/project/{id}` deep link + clipboard handoff, triggered by the Approval Poller. The composed prompt is now also written to the Sheet as a durable fallback record, rather than included in a Telegram message.

**Resolved this session (keyword false-positive audit, triggered by real jobs the user flagged as irrelevant on Beacon)**:
- Three distinct root causes found and fixed: (1) "X12" colliding with healthcare shift-count notation ("Weekx12", "x12-Hour Shifts") — tightened the regex twice rather than removing the keyword, since it has hundreds of genuine EDI matches; (2) a tracked company's own name also being a `tech_keyword_include` entry (Kong, Boomi) — every posting from that company trivially matched its own boilerplate mention of its own name, fixed by excluding self-referential matches in `evaluate_job`; (3) `require_us_location` missing ~15 real non-US cities (Dublin, Paris, Bengaluru, etc.) plus an entire unhandled pattern (ISO-3166 country-code location prefixes like "AE-Dubai") in `app/location.py`
- Separately, and NOT the same bug class: "Integration"/"Messaging" as bare `tech_keyword_include` entries are simply too generic — they match almost any corporate boilerplate at a company with a large, organizationally-diverse open-role catalog (confirmed at Stripe, Kong, and especially OpenAI, which has 719 open roles and was flagged at seed time to "expect heavy keyword filtering"). **Decided to keep both keywords rather than remove them** — 218 of 1,264 Beacon jobs (17%) depend on them, and while most is noise, some genuine messaging-infrastructure titles (that don't mention a specific product name) would be lost too. `title_exclude` (now ~50 terms) is the accepted trade-off: safer, doesn't risk losing real matches, but doesn't scale cleanly to a company the size of OpenAI — expect this class of report to recur
- Found and fixed a genuine, unambiguous coverage gap along the way: `role_keyword_include` had "Solutions Architect" (plural) but not "Solution Architect" (singular) — added it, immediately surfacing several already-relevant jobs (Amazon, TD Synnex) that had only been passing by accident via the generic tech keywords
- 116 already-live postings across all of this were manually evicted from Beacon to the Job Log during the same session

**Resolved the following session (RN/Nurse recurrence — root-caused a systemic gap, not a new bug)**:
- The `require_us_location` fix above had its own bug: "South Paris, Oxford County" (a real Maine town) was misclassified as Paris, France, because "paris" was checked unconditionally before any US signal — the same real-namesake risk already known for Manchester/Melbourne/Geneva applies to Paris, Dublin, London, Berlin, Warsaw, Madrid, and Lisbon too. Restructured `app/location.py`'s `_segment_is_us` into a two-tier check: unambiguous non-US names first (preserves the Herzliya/Illinois fix), then a US confirmation (state abbreviation/name or a "County"/"Parish" suffix), then the risky namesake-colliding names — the two orderings can't be satisfied by a single pass, hence two tiers
- Root-caused why "already fixed" bad jobs kept reappearing in reports: `run_filter_engine` never re-checks a job already on Beacon against a later criteria change — it only decides whether to *add* new `status='new'` jobs. A full audit of the live backlog found 67 stale jobs (31 aged past `posted_within_days`, 36 failing current keyword/location criteria) that predated the relevant fixes and were never swept up. Evicted the 36; the 31 posted-date ones and whether to build automatic re-validation into the pipeline are still open
- Self-audit (a new systematic query cross-checking every `title_exclude` term against every job that also matches a `role_keyword_include` term) caught 5 false exclusions this session's own `title_exclude` additions had introduced: "Digital Natives", "Communications" (collided with "Telecommunications" and an industry-vertical usage), "Robotics", "Supply Chain", and "Payroll" were all narrowed to their literal original targets. - Enrichment moved from fully-manual to a scheduled job (per direct request, "run it on schedule keeping api quota in mind") — `run_scheduled_enrichment()`, initially once/day at 7pm, capped at a new `daily_enrichment_limit` filter_setting (default 100). The cap is sized around **FMP's confirmed 250-requests/day free-tier limit** (checked against FMP's actual published pricing, not assumed) — StartupHub.ai has no published limit, so the cap follows FMP's harder, confirmed constraint instead of guessing at StartupHub's

Offset fit-scoring to 5 minutes after the main pipeline (8:05/1:05/6:05); enrichment initially got its own distinct off-peak time (7pm) for the same reason

- Fourth item, discovered draining the new StartupHub backlog rather than requested: the first uncapped run crashed on `gspread.exceptions.APIError [503]` from `find_existing_row` mid-batch, since `app.sheets_retry.call_with_retry` only retried 429, not Google's other transient 5xx codes — widened to match `app.http_client.RETRYABLE_STATUS_CODES` (`{429, 500, 502, 503, 504}`), and each company's `_push_to_beacon` call in both enrichment passes is now wrapped in its own try/except so one Sheets failure logs and continues rather than aborting the whole run. Re-running afterward surfaced a second, unrelated, **pre-existing** bug: `app.fmp.search_symbol()` had no name-verification step, so FMP's fuzzy name search could return a completely unrelated company for short/generic names -- invisible with only 7 companies ever checked before this session, but corrupted `company_type`/`funding_stage`/`revenue_or_valuation`/`industry`/`hq_location` for ~15 companies (Saic→Mosaic's ticker, Bloomberg→a gold-mining stock, Google→an unrelated micro-cap, plus RSM/SHI/LMI/ECS/MPD/IDEX/Sage/Sentinel/Healthie/AeroTEC/"Executive Search"/"Technology Company") once FMP started running at real volume (~250 companies/day). All 15 corrupted companies were fully reset (confirmed none had legitimate StartupHub data blocked underneath, since StartupHub ran first for all of them and found nothing) and their Beacon rows re-pushed blank
- Separately noticed (not investigated, flagged as a background task): `scheduler.log` shows dozens of repeated Anthropic API `400 Bad Request` errors during the 2026-07-05 00:02 pipeline run, likely from visa-scan or fit-scoring -- a 400 means a malformed request, not a transient rate-limit issue, so it's probably silently failing to process a batch of jobs rather than retrying successfully
