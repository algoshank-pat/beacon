# Job Search Automation App — Build Spec for Claude Code

## Project Goal
Build a personal job search automation tool (Python backend, SQLite DB, Google Sheets for tracking, approval, AND notification) that:
1. Polls job boards on a schedule and ingests new postings — both broad keyword discovery and targeted company tracking
2. Filters postings against my live-editable criteria (title, seniority, location, salary, company priority, posted-date window)
3. **The moment a job passes filtering, it gets a row on the Beacon sheet — no waiting on visa-scan or fit-score.** This was a deliberate pivot from the original score-gated design (below): the priority is seeing what's out there as fast as possible, not waiting on the expensive/rate-limited scoring step. Score and visa columns start blank and fill in as those steps complete
4. Visa scanner updates that row's Visa Flag in place; if restricted (and `require_visa_sponsorship` is on), the row is evicted from Beacon and moved to the Job Log ("Filtered") sheet instead
5. Fit-review score (JD vs. my resume) runs on its **own separate schedule** (same three daily times as the main pipeline, but not bundled with ingestion/filtering — used to run overnight instead) using the Claude API, only for jobs I've flagged "Go Score" on Beacon — updates the Beacon row's score in place if it crosses `fit_score_threshold`, evicts it to the Job Log if not
6. I record my decision (Approve/Deny) directly in the Beacon sheet's Decision column — this also drives row color (green/red) so decided jobs are visually distinct at a glance. The app polls for this on a schedule and acts on it (Approval Poller — **built**, see Approval via Google Sheets below; the actual Claude Desktop handoff on Approve is still M7, not built — today Approve just records the decision)
7. On Approve, hands off resume/cover letter generation to my existing "Job Search" project in Claude Desktop (not a direct API call), re-runs the fit score once I register the result, and updates the Sheet
8. Everything downstream of that (application status, salary, interview notes) lives in the Sheet, hand-maintained

**No Telegram, no bot, no webhook.** Notification is handled by **Google Sheets' own native notification feature** (Tools → Notification rules, set to "Any changes are made" → "Email right away") — I enable this once, myself, in the Sheets UI. Every time the app writes anything to the Sheet (a new match, a reminder, a status update), Google emails me automatically. **Resolved (was ambiguous in earlier drafts of this doc): the app authenticates via a service account**, a distinct Google identity from `algoshank@gmail.com`. That distinctness is exactly what makes native notifications work — Google emails me on the service account's writes but not on my own manual edits. **Only the first half is confirmed live** (app write → email arrived, tested during M5); the second half (my own manual edit → no self-notification) hasn't been independently confirmed yet — worth a quick check, though if it turns out my own edits do notify too, that's noisier than ideal, not broken.

**Scope is intentionally trimmed and now further simplified.** My effort mostly goes to (a) fine-tuning my resume and (b) finding new jobs matching my criteria. Application/interview tracking is minor effort for me and a spreadsheet already handles it well.

This is a **workflow**, not an autonomous agent: fixed sequence, LLM called at specific steps only. Approval — via the Sheet's Decision column — is a hard gate before any resume-generation handoff or external action.

**Non-negotiable boundary**: this app never submits an application, ever, under any configuration. It stops at handing off resume/cover letter generation and updating the Sheet. I always review and submit manually. This boundary should not be relaxed later without an explicit, separate decision.

## Tech Stack
- **Backend**: Python 3.14 (already installed on Windows native — if any library fails to install due to missing 3.14 wheels, fall back to Python 3.12 for this project; install it alongside 3.14 with "Add to PATH" unchecked), FastAPI (settings UI only, not built yet), APScheduler (`apscheduler==3.11.3`) running three internal jobs inside one long-lived process (main pipeline — including a Seed-via-Sheet onboarding step ahead of ingest — fit-scoring at the same three daily times, and the Approval Poller — all three built)
- **DB**: SQLite — pipeline operational state only, not tracking data. **WAL mode enabled** (`PRAGMA journal_mode=WAL` at connection setup) plus **`PRAGMA busy_timeout=30000`** (30s) — necessary because CLI commands get run manually and independently, sometimes overlapping each other or the scheduler process (a real scenario hit live: `filter` and `enrich-companies` run in separate terminals at the same time). WAL alone only helps a reader coexist with a writer; `busy_timeout` is what lets a second *writer* wait its turn instead of failing immediately with "database is locked." **The deeper fix, also required**: every DB write must be `commit()`-ed *before* any Sheets API call, never left open across one — Sheets calls can retry/sleep for minutes under quota pressure (see `sheets_retry` below), and an open SQLite write transaction held that long starves any other process trying to write, no `busy_timeout` value would be enough. Hit and fixed live in `filter_engine.py`/`visa_scan.py`/`fit_scoring.py`/`sheets.py`
- **Tracking + Approval + Notification**: Google Sheets, via `gspread` (`gspread==6.2.1`) using **service account authentication** (`gspread.service_account(filename=...)`), credentials at `GOOGLE_SHEETS_CREDENTIALS_PATH`. The service account is a distinct Google identity from `algoshank@gmail.com` and must be shared as an Editor on both spreadsheets — this distinctness is what's supposed to make native Sheets notifications fire on app writes but not on your own manual edits. App-write → email confirmed live during M5; self-edit → no-email not yet independently confirmed
- **Sheets write-quota resilience**: `app/sheets_retry.py` wraps every Sheets write/read call used by the high-volume writers (Beacon, Job Log) with retry-and-backoff on `gspread.exceptions.APIError` code 429 — hit live during the first bulk backlog run (logging every passing/filtered job in a tight loop exceeds Google's per-minute write quota well before a 1,000+ job backlog finishes). Non-429 errors and exhausted retries still raise
- **Timezone handling**: `tzdata==2026.2` — Windows doesn't ship the IANA timezone database that Python's stdlib `zoneinfo` needs, so `ZoneInfo("America/Chicago")` raises `ZoneInfoNotFoundError` without this package installed. All Sheet-facing dates render as `mmddyyyy HH:MM` in Central time (`app/dates.py`), converted from whatever format/timezone the source produced
- **LLM**: Anthropic API (`claude-sonnet-5` for fit scoring, `claude-haiku-4-5` for cheap visa classification on ambiguous cases only)
- **Resume/cover letter generation**: Claude Desktop, via its `claude://claude.ai/project/{id}` deep link — NOT an API call
- **Notifications**: Google Sheets' own native notification feature (Tools → Notification rules) — enabled once in the UI, no app code. If the Approval Poller fails to reach the Sheets API, consecutive failures are logged to `workflow_runs`/`step_logs` and surfaced via `python -m app.cli last-run` — no separate alert channel needed

**Runtime environment**: build inside a dev container, but the app must **run natively on your Windows host** — the Claude Desktop handoff requires `os.startfile()`/clipboard access, unavailable in a container.

**Single persistent process, not separate scheduled tasks**: `app/scheduler.py` runs a `BlockingScheduler` with four internal jobs — main pipeline (seed-via-sheet → ingest → filter → visa-scan, 3x/day at fixed times, default 8am/1pm/6pm via `CronTrigger`), fit-scoring (its own separate `CronTrigger`, 5 minutes after each of those same three times — used to run overnight at 2am, then moved to the same instant as the main pipeline, then offset by 5 minutes once same-instant firing turned out to cause real Sheets-quota contention), company enrichment (its own `CronTrigger`, 3x/day at 8:10am/1:10pm/6:10pm — 10 minutes after each main pipeline run; each firing runs two independent passes, StartupHub uncapped and FMP capped by cumulative daily usage; used to be fully manual, then once/day at 7pm as a single combined pass, then 3x/day, then split into these two independently-paced passes — see Company Enrichment Data Sources), and the Approval Poller (every `approval_poll_interval_minutes`, default 30, via `IntervalTrigger`) — avoids SQLite cross-process file-locking risk on Windows. **Why fit-scoring and enrichment are their own scheduled jobs rather than chained into the main pipeline**: a failure/slowdown in either must never block ingest/filter/visa-scan; fit-scoring originally also ran overnight specifically to get a fresh, uncontested daily token budget instead of competing with same-day filter/visa-scan spend, but that isolation stopped mattering once scoring became manually-gated by the Beacon sheet's "Go Score" flag rather than automatic across the whole backlog. **Real bug found and fixed**: fit-scoring firing at the *exact same instant* as the main pipeline (both at 8/1/6 o'clock sharp) let APScheduler's thread-pool executor run them concurrently, both hitting the Sheets API at once -- a real contributor to the 429 quota errors hit repeatedly this session. Offsetting fit-scoring by 5 minutes, and enrichment by 10 minutes, avoids this. Jobs land on Beacon immediately at filter-pass time rather than waiting on a score to compute — see Fit Scoring Logic below. Windows Task Scheduler's only job is keeping the process alive: a single "at startup" trigger with restart-on-failure enabled — this is the actual answer to "how do I get back to this after a restart," documented step-by-step in `RUNBOOK.md`. **Attempted to register this from within Claude Code (`scripts/register-scheduler-task.ps1`, `Register-ScheduledTask`) and it fails with "Access is denied"** — this environment's Claude Code session doesn't have permission to register scheduled tasks; it has since been registered successfully by hand from a normal PowerShell window.

**Single-instance lock, added after a real incident**: a single `python -m app.scheduler` launch was observed more than once to result in two independent `BlockingScheduler` loops running concurrently against the same DB/Sheet — confirmed via overlapping `workflow_runs` rows (not just a process-list artifact), causing duplicate pipeline runs and duplicate Sheets writes. Root cause was never conclusively identified (reproduced even from a single, deliberate, manually-typed launch in a fresh terminal) — could be specific to this machine's venv/Python install. Rather than keep relying on manually checking `Get-Process`/`Get-CimInstance` before every launch, `app/scheduler.py` now acquires an OS-level exclusive lock on `scheduler.lock` at startup (`msvcrt.locking`, Windows-only) and refuses to run — exiting with a clear log line — if another instance already holds it. The lock releases automatically when the holder's process exits, even on an ungraceful `Stop-Process -Force`, so there's no stale lock file to manually clean up.

**Process resilience — catch-up on sleep, crash, and reboot**:
- The main pipeline job fires once immediately on every process start (in addition to its regular schedule) — configured with `next_run_time=datetime.now()` at registration. This matters more now than it would with frequent polling: with only 3 fixed runs/day, sleeping through the 1pm slot means nothing happens again until 6pm without this catch-up
- `misfire_grace_time` set generously (6 hours) on both scheduled jobs so a run missed during laptop sleep fires when the process next gets a chance, rather than being silently skipped by APScheduler's short default grace window
- Ingestion loses nothing to downtime regardless — sources return all currently open postings on every poll, not an incremental feed, so downtime only delays discovery
- Fit-scoring naturally catches up over consecutive nights too — already-scored jobs are skipped, unscored ones roll over to the next run, throttled only by `daily_token_budget`
- Practical option: set Windows power settings to avoid sleep during your active hours if you want same-day discovery of overnight postings — a convenience preference, not a correctness requirement given the catch-up logic above

## Database Schema (SQLite — pipeline state only)

```sql
-- Companies I'm tracking (seed list I maintain + auto-added from postings)
CREATE TABLE companies (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL UNIQUE,
    board_url TEXT,
    source_type TEXT,              -- 'greenhouse' | 'lever' | 'ashby' | 'smartrecruiters' | 'adzuna' | 'manual'
    industry TEXT,
    company_size TEXT,
    priority_tier TEXT,            -- 'S' | 'A' | 'B' | 'C'
    is_favorite BOOLEAN DEFAULT 0,
    employee_count INTEGER,
    employee_count_range TEXT,
    founded_year INTEGER,
    hq_location TEXT,
    linkedin_company_url TEXT,
    famous_product TEXT,
    visa_sponsorship_history TEXT, -- 'known_sponsor' | 'known_non_sponsor' | 'unknown' (posting-language signal)
    h1b_sponsor_last_5yrs BOOLEAN,
    h1b_petitions_last_5yrs INTEGER,
    h1b_data_last_checked TIMESTAMP,
    company_type TEXT,             -- 'public' | 'private'
    funding_stage TEXT,            -- 'bootstrapped' | 'seed' | 'series_a' | 'series_b' | 'series_c' | 'series_d_plus' | 'ipo_public' | 'unknown'
    revenue_or_valuation TEXT,     -- free text: "$50M ARR (est.)" or "$12.4B market cap"
    revenue_valuation_source TEXT,
    financial_data_last_checked TIMESTAMP,
    notes TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Every job posting ingested
CREATE TABLE jobs (
    id INTEGER PRIMARY KEY,
    company_id INTEGER REFERENCES companies(id),
    title TEXT NOT NULL,
    url TEXT NOT NULL UNIQUE,
    apply_url TEXT,
    description TEXT,
    location TEXT,
    remote_type TEXT,
    seniority TEXT,
    job_function TEXT,
    salary_min INTEGER,
    salary_max INTEGER,
    salary_source TEXT,
    adzuna_salary_min INTEGER,      -- Adzuna's algorithmic estimate, captured once at ingest and never
    adzuna_salary_max INTEGER,      -- overwritten (added in migration 0003) -- see Sheet columns table below
    visa_flag TEXT,
    visa_snippet TEXT,
    duplicate_of_job_id INTEGER REFERENCES jobs(id),
    posted_at TIMESTAMP,
    first_seen_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    notified_at TIMESTAMP,          -- set when Sheet row is first appended; drives stalled-decision reminders
    decision_processed_at TIMESTAMP,-- idempotency guard: once set, a decision has been acted on and won't be reprocessed
    status TEXT DEFAULT 'new',      -- 'new' | 'filtered_out' | 'scored' | 'notified' | 'approved' | 'rejected' | 'closed' | 'duplicate'
    rejection_reason TEXT,
    sheet_row_number INTEGER,
    source_type TEXT                -- 'greenhouse' | 'lever' | 'ashby' | 'smartrecruiters' | 'adzuna' (added in migration 0002, not the original DDL) —
                                     -- needed to correctly scope closed-job detection and cross-source fuzzy dedup per source, since
                                     -- a company can get rows from both Adzuna broad discovery and its own targeted ATS board
);

-- One row per fit-scoring event (initial score + post-resume re-score)
CREATE TABLE fit_scores (
    id INTEGER PRIMARY KEY,
    job_id INTEGER REFERENCES jobs(id),
    resume_file_path TEXT,
    score INTEGER,
    gap_analysis TEXT,
    scored_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- My notes on generated resumes — included in future Claude Desktop handoff prompts
CREATE TABLE resume_feedback (
    id INTEGER PRIMARY KEY,
    job_id INTEGER REFERENCES jobs(id),
    feedback TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Live-editable filter criteria (replaces static config.yaml at runtime)
CREATE TABLE filter_settings (
    id INTEGER PRIMARY KEY,
    key TEXT NOT NULL UNIQUE,
    value TEXT,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE filter_keywords (
    id INTEGER PRIMARY KEY,
    category TEXT NOT NULL,
    keyword TEXT NOT NULL,
    is_active BOOLEAN DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(category, keyword)
);

-- Observability: every run of the polling/filter/score/notify/decision pipeline
CREATE TABLE workflow_runs (
    id INTEGER PRIMARY KEY,
    run_type TEXT NOT NULL DEFAULT 'main_pipeline', -- 'main_pipeline' | 'approval_poll'
    started_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    finished_at TIMESTAMP,
    status TEXT,
    jobs_ingested INTEGER DEFAULT 0,
    jobs_filtered_out INTEGER DEFAULT 0,
    jobs_scored INTEGER DEFAULT 0,
    jobs_notified INTEGER DEFAULT 0,
    decisions_processed INTEGER DEFAULT 0,
    tokens_used_input INTEGER DEFAULT 0,
    tokens_used_output INTEGER DEFAULT 0,
    estimated_cost_usd REAL DEFAULT 0,
    error_summary TEXT
);

CREATE TABLE step_logs (
    id INTEGER PRIMARY KEY,
    workflow_run_id INTEGER REFERENCES workflow_runs(id),
    job_id INTEGER REFERENCES jobs(id),
    step_name TEXT,                 -- 'ingest' | 'filter' | 'beacon_add' | 'visa_scan' | 'fit_score' | 'sheet_sync' | 'decision_poll' | 'reminder'
                                     -- ('notify' retired post-M5 -- fit_score/visa_scan now update or evict the Beacon row directly, see §"Fit Scoring Logic")
    step_status TEXT,
    detail TEXT,
    tokens_input INTEGER DEFAULT 0,
    tokens_output INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**What's deliberately NOT in the database**: `applications`, `interview_rounds`, `cover_letters`, `resumes` as relational tables. That tracking lives entirely in the Google Sheet.

**Migrations**: schema changes ship as numbered `.sql` files in `app/migrations/`, tracked in a `schema_migrations` table, applied via `python -m app.cli migrate`. The DDL above is migration `0001_initial.sql`; `source_type` on `jobs` shipped as `0002_add_jobs_source_type.sql`; `adzuna_salary_min`/`adzuna_salary_max` shipped as `0003_add_adzuna_salary_estimate.sql`. Don't hand-edit `0001` after it's been applied anywhere — add a new migration instead.

## Company Enrichment Data Sources

**Automated, not manual entry** (superseding earlier drafts of this doc, which assumed manual entry or a single enrichment API). Runs both on-demand (`python -m app.cli enrich-companies`) and **on its own schedule** (`run_scheduled_enrichment`, 3x/day at 8:10am/1:10pm/6:10pm, added per direct request -- "run it on schedule keeping api quota in mind" -- and later moved from an initial once/day cadence to this 3x/day one per a direct follow-up request, "Can we invoke enrichment more than once a day?"), scoped to companies with at least one job **currently on the Beacon sheet** (`jobs.sheet_row_number IS NOT NULL` — not every company that merely passed keyword+location filtering during broad discovery, which can run into the hundreds). This scoping changed from an earlier `status='notified'` check once jobs stopped being gated behind a score to reach Beacon (see Fit Scoring Logic) — `sheet_row_number` is now the live signal for "is this job something the user is actually looking at," independent of `jobs.status`.

**Scheduled runs are capped at `daily_enrichment_limit`** (a new filter_setting, 200 companies/day as of this session, raised from an initial 100 -- see below) to respect **FMP's confirmed 250-requests/day free-tier limit** (checked against FMP's actual published pricing page, not assumed) -- each company costs 1 FMP call (private, no ticker match, the common case) or 2 (publicly traded: search + profile). StartupHub.ai has no published rate limit, so the cap follows FMP's harder, confirmed constraint rather than guessing at StartupHub's. 571 companies with a job on Beacon had never been enriched when this shipped -- expect a few days to drain that backlog.

**Cap is enforced cumulatively across the day, not per-run**, which is what makes running the FMP pass 3x/day (rather than just once) safe. `app.enrichment.get_fmp_enriched_today_count()` counts companies whose `financial_data_last_checked` falls today (same pattern `app.budget.get_spend_today` uses for the LLM token budget); `run_scheduled_enrichment` computes `remaining = daily_enrichment_limit - get_fmp_enriched_today_count()` at the start of every run and passes that as the FMP pass's limit, so a run after the backlog's been mostly drained today just claims whatever's left rather than blindly requesting another full batch. Once the day's cap is reached, the FMP pass is skipped for that run (`fmp_skipped: True` in the result) while the StartupHub pass still runs.

**`daily_enrichment_limit` raised from 100 to 200/day** (per direct follow-up request -- "would I exceed quota... any enrichment which is free and has no quota, run it more frequently"): confirmed 3x/day at 100/day cannot exceed FMP's quota regardless, since the cumulative cap already bounds the day's total (worst case 200 of 250 calls). Live data at the time showed 0 of the first 7 enriched companies were public matches (the 2-call case), so real cost runs closer to 1 call/company than the assumed worst case -- meaning 100/day only used ~40% of FMP's real budget even worst-case. 200/day keeps the same safety property (worst case 400 calls only if every company were public, which essentially never happens for this startup-heavy company list; realistic usage ~200-220/day, still under 250) while draining the FMP backlog roughly twice as fast.

**Enrichment split into two independent passes with independent "checked" tracking** (per direct follow-up request -- "try searching for every company using startuphub.ai and then leave the rest to fmp... since it's free, we can just hit this API more times"): reordering the two sources alone was a dead end and confirmed as such before building anything -- FMP is the *only* source for `employee_count`/`company_type`/`funding_stage`/`revenue_or_valuation`, so every company still needs an FMP query regardless of which source is tried first; call order never reduces FMP's call volume, only decoupling the two sources' cadence does. Migration `0004_add_startuphub_last_checked.sql` adds `startuphub_last_checked` alongside the existing `financial_data_last_checked`, so the two sources now each track their own "already checked" state independently. `app/enrichment.py` exposes `run_startuphub_enrichment()` (no `limit` by default -- StartupHub.ai has no published rate limit, so it processes the entire never-checked-by-StartupHub backlog every time it runs) and `run_fmp_enrichment()` (capped by the caller, since FMP is the one source with a real, confirmed quota) as separate functions; the scheduler calls both directly with their respective caps, while `run_enrichment()` remains as a combined wrapper used only by the manual `enrich-companies` CLI command (applying the same `--limit` independently to each pass). **Confirmed live, immediately**: Booz Allen Hamilton, already FMP-matched earlier in the session (a confirmed public company), got `founded_year=1917` filled in by its very first StartupHub check under the new design -- a field FMP's own `/profile` endpoint never exposes at all (see source 1 below), so every public company was permanently missing it under the old "skip StartupHub entirely once FMP matches" rule. 576 companies were eligible for the StartupHub-only backlog at the time of this change (more than the FMP-only backlog, since every company FMP had already checked became newly eligible for a StartupHub check too).

**Draining that backlog surfaced two more real bugs, not requested but found along the way.** First: the initial uncapped StartupHub run crashed mid-batch on a Google Sheets `503` from `find_existing_row` -- `app/sheets_retry.py`'s `call_with_retry` only ever retried HTTP `429`, not Google's other transient 5xx codes, even though `app/http_client.py` already retries the full `{429, 500, 502, 503, 504}` set for every other API call in this pipeline. Widened `call_with_retry` to match, and wrapped each company's `_push_to_beacon` call inside both enrichment passes in its own try/except, so one Sheets hiccup logs and moves on instead of aborting the rest of a large batch. Second, more serious: re-running after that fix processed ~250 FMP companies in one day (versus 7 total before this session), which exposed a **pre-existing** bug in `app/fmp.py`'s `search_symbol()` -- it had no name-verification step at all, just trusting FMP's fuzzy `/search-name` endpoint's first result. Confirmed live: "Saic" (a real public company, ticker `SAIC`) matched Mosaic's ticker instead, "Bloomberg" matched an unrelated gold-mining stock, "Google" matched an unrelated micro-cap, and about a dozen more (RSM, SHI, LMI, ECS, MPD, IDEX, Sage, Sentinel, Healthie, AeroTEC, plus two clearly-bad seed-list entries -- "Executive Search" and "Technology Company" -- that never should have matched anything) got `company_type`/`funding_stage`/`revenue_or_valuation`/`industry`/`hq_location` populated with a completely unrelated company's data. Fixed with a new `_names_plausibly_match()` check (word-overlap after stripping corporate suffixes like Inc/Corp/LLC) that `search_symbol` now requires before trusting any candidate -- deliberately not acronym-aware (an earlier attempt to make "Saic" match "Science Applications International Corp" via acronym expansion was reverted, since it risked reintroducing this exact bug for other short 2-4 letter names like RSM/LMI/ECS/MPD/SHI). This is consistent with the pipeline's existing "leave it blank rather than risk being wrong" enrichment philosophy: losing a legitimate acronym match is a far smaller cost than another wrong one slipping through undetected. All ~15 already-corrupted companies were reset (every enrichment field plus both `financial_data_last_checked`/`startuphub_last_checked` nulled -- confirmed via timestamps that StartupHub had run first for every one of them and found nothing, so there was no legitimate data being thrown away) and their Beacon rows re-pushed blank.

**Enrichment must always be free, no exceptions — built.** The third source below (Claude Sonnet 5 + web search) has been removed entirely — if the two free sources don't have a company's data, the field just stays blank for the user to fill in manually later ("if that info is not available, do not spend tokens on it — if not, that's fine, I'll update them when I can"). `enrich-companies` no longer even requires `ANTHROPIC_API_KEY`. Two sources remain, run as two independent passes (see above), each merging via `COALESCE` so neither overwrites a field the other already found:

1. **Financial Modeling Prep** (`financialmodelingprep.com`, free tier, `FMP_API_KEY`) — **publicly traded companies only**; their `/profile` endpoint requires a resolvable stock ticker, so private companies simply return no match. `GET /stable/search-name?query=<name>` resolves name → ticker (preferring NASDAQ/NYSE when a company is dual-listed), then `GET /stable/profile?symbol=<ticker>` returns `employee_count`, `hq_location`, `industry`, `company_type='public'`, `funding_stage='ipo_public'`, `revenue_or_valuation` (market cap). Does **not** expose `founded_year` at all -- that gap is exactly why the StartupHub pass still runs against already-FMP-matched companies (see above).
2. **StartupHub.ai** (`startuphub.ai`, free tier, `STARTUPHUB_API_KEY`) — an AI-startup directory; covers a meaningful slice of private/VC-backed companies, but **not universal** (a small non-AI staffing/consulting shop returned no match in testing). `GET /startups?q=<name>` — **the `q=` parameter is the only one that actually searches**; `query=`/`name=`/`search=` are silently ignored by their API and return an unrelated default listing (confirmed live: querying `name=Workato` returned Medtronic as the top result). The returned result's name is verified against the query before use, since even `q=` search can surface unrelated results further down. On the current (free) key tier this only returns `hq_location`, `founded_year`, and `industry` — **not** `employee_count`/`funding_stage`/`revenue_or_valuation`, which require a paid "Pro Lite or higher" plan for their `/enrich` endpoint (confirmed via a live 403).

~~3. Claude Sonnet 5 + server-side web search~~ — **removed per direct request.** Previously filled whatever was still missing after 1 and 2 at a real cost of ~$0.15–0.75/company (the most expensive per-unit LLM cost anywhere in this pipeline). `enrich_company()`/`EnrichmentParseError`/`ENRICHMENT_PROMPT`/`SONNET_MODEL` are gone from `app/enrichment.py` entirely — `run_enrichment()` no longer takes a `client` param at all. **Known consequence, accepted**: a meaningful fraction of companies (private ones StartupHub doesn't cover) will now simply never get Employee Count/Funding Stage/Revenue automatically — they stay blank until filled in by hand.

Adzuna's API was checked directly (not assumed) and confirmed to expose nothing beyond a company `display_name` — no firmographic data at all, so it was never a viable source.

`h1b_sponsor_last_5yrs`/`h1b_petitions_last_5yrs` — public DOL LCA disclosure data, a distinct dataset none of the above sources reliably reproduce. Still a manual/future path, not automated. **Clarified this session**: the Visa Scanner (see Visa Sponsorship Detection Logic) never reads or writes these columns — it only classifies what a specific job posting's own text says about sponsorship, which is a completely different signal from a company's real historical H-1B filing record. These columns exist in the schema and sit empty. Real data does exist publicly (DOL's LCA disclosure data, USCIS's H-1B employer data hub, aggregated more usably by third-party sites like h1bdata.info/myvisajobs.com) but integrating it would be a distinct feature, not an extension of the current text classifier — marked as a **future item**, not being built now. In the meantime, the user-curated "real H-1B sponsors" list added to `seed_companies.yaml` this session is a better real-world signal of historical sponsorship than anything currently automated.

**"Location State" column** was added after a direct request ("I want to add Location state column... I cannot figure out which state based on county and city names"). The real obstacle: a county name alone never determines a state -- "Washington County" exists in about 30 states -- and the scraped `location` text is inconsistent across every source (city+abbreviation, city+full state name, state+"United States", city+county with no state at all, bare city names, ATS-specific shorthand like "US-CA-Menlo Park", semicolon-separated multi-office postings, and every phrasing of "Remote"). Rather than guess at how to handle the hard county case, three options were laid out and the user picked: bundle a small static reference dataset (public-domain US Census county and place-population data, fetched directly from census.gov) instead of adding the `uszipcode` package's ~9 transitive dependencies or just leaving county-only rows unresolved. New `app/location_state.py` resolves state > county (disambiguated by a sibling city token, population-tie-broken when two states share both a county name and a same-named city) > bare city name (best-guessed to the most populous nationwide match -- "Austin" alone is almost certainly Austin, TX). Any remote/nationwide/multi-state signal maps to `"Remote-USA"`; genuinely unresolvable strings are left blank rather than guessed. One real near-miss caught before it caused damage: the column was originally going to be inserted right next to "Location", which would have silently misaligned every following column against ~3,190 rows of live sheet data, since this sheet's writes are positional and the schema's own docstring already warned that reordering needs a full rebuild -- appended at the end of the column list instead, and the header + all existing Beacon rows were backfilled live via one batched write. Confirmed coverage against the real backlog: 91.4% resolved, the rest left honestly blank (informal neighborhood names, non-US remote postings) rather than silently wrong. **The user then asked for it moved anyway** ("Location state has to be right next to Location (column E)") -- this time it genuinely required moving live columns on both sheets (insert a new column at F pre-filled with the existing values, delete the now-duplicate trailing one), not just reordering a Python list, since real data already sat in fixed positions. That surfaced two real bugs in the process rather than going smoothly: the column-move library call shifted every data row correctly but left the header row stale on both sheets (caught by reading back several rows spread across the full range, not just the top of the sheet, and comparing against the header); and, while double-checking that fix, a second, unrelated, pre-existing bug -- the Job Log sheet's conditional-format rules had never been deleted before being re-added on a header change, so this session's two header changes had silently stacked 12 duplicate rules where there should be 3. Both fixed, and the whole thing re-verified with a random sample of rows checked directly against the database, not just eyeballed on the sheet. **The user then reported something much bigger** ("After this change, lots of issues with the shifting of cells"). The real cause turned out to have nothing to do with the column-move code itself: a persistent background process for this pipeline (`app.scheduler`, running continuously as a Windows Scheduled Task since before any of this session's Location State work) had the *original* column layout loaded in memory the whole time -- Python doesn't reload code in a running process, so it never picked up either of this session's schema changes despite both being live on disk. Every write that process made kept using its old, now-wrong column positions, landing values in cells the header no longer agreed with. Confirmed the exact pattern live (a salary value sitting in the "Date Posted" cell, a date sitting in "Salary Range") before touching anything, then fixed it in the right order: restarted the actual process first (so the bleeding stops), then repaired the already-damaged rows precisely -- recomputing every column the database can reliably reconstruct and only overwriting cells that were actually wrong, while never touching the columns that only exist as sheet state (Decision, My Decision, reminder timestamps, fit scores) since those can't be safely reconstructed and there was no evidence any of them had actually been touched. 674 rows needed 1-2 cells fixed each; verified by re-checking all 3,363 Beacon rows afterward with zero mismatches left.

**Real, live data-loss bug found and fixed while adding Birlasoft**: `app.seed.seed_companies` (the `seed_companies.yaml` loader) blindly overwrote `employee_count`/`company_type`/`hq_location`/`founded_year`/`revenue_or_valuation`/`industry`/etc. back to `NULL` for *every* already-enriched company also listed in the yaml file on every re-run -- since that file never sets those fields at all (they're filled in later, entirely separately, by `app.enrichment`), the old blanket `UPDATE ... SET col = ?` for every column clobbered them with `None` regardless. This had been silently happening on every `seed-companies` re-run all session; it only surfaced now because re-running it to add one new company (Birlasoft) immediately wiped real FMP/StartupHub data for Boomi, Workato, n8n, and Twilio, caught by checking their fields right after. Fixed by switching to `COALESCE(?, column)` per field -- the same merge pattern `app.enrichment._apply_enrichment` already uses for exactly this reason -- so an explicit yaml value still overrides, but a field the yaml never touches survives untouched. The 4 affected companies were restored not by re-fetching (which risks a different/wrong match) but by replaying their complete chronological history of successful enrichment attempts recorded in `step_logs`, applying the same first-non-null-wins COALESCE logic in Python to reconstruct their exact pre-bug state.

**That same investigation surfaced a second, separate, deeper limitation, left unfixed**: a live FMP re-fetch for "Kong" (the tracked API-gateway company) during this recovery matched an unrelated Beijing-listed utility company, ticker `0295.HK`, industry "Renewable Utilities" -- confirmed by checking the raw FMP response. This is NOT the same bug as the earlier "Kong ⊂ Hong Kong" description-text collision (see Filter Criteria notes below) or the FMP name-verification fix from earlier this session (`_names_plausibly_match`, which rejects a fabricated/unrelated name) -- this is a genuine collision between two *real* companies that happen to share the literal name "Kong" in FMP's own database, which no amount of name-string comparison alone can resolve. Left Kong's enrichment fields blank rather than restore or retry, since a retry would likely surface the same wrong match again; would need a second disambiguating signal (industry, HQ country) that `search_symbol` doesn't currently have access to -- not built.

**Also fixed in the same session, structural rather than a one-off retry**: `app.sheets.get_main_worksheet` and `app.job_log.get_job_log_worksheet`'s initial `client.open_by_key()` call now goes through `call_with_retry`, same as every other Sheets API call in both modules -- this exact call had failed on a transient 429 or 503 several separate times across this session (most recently mid-restore, moments after a heavy write burst), always requiring a manual retry since it was the one call in each module that had never been updated to match the rest.

**Real-world consequence of this growth, surfaced during a "why is `filter` taking hours" review**: a single `ingest` run against the expanded targeted-tracking list pulled in 6,538 new jobs in one shot, on top of an earlier 4,643-job jump from the first batch — pushing the `filter` backlog past 12,000 jobs. Every job `filter` evaluates costs ~2-3 Sheets API calls today (a full-column duplicate-check read, plus an append or Job Log upsert) regardless of whether it passes — so Sheets write-quota throttling, not token cost, is what makes a big backlog slow. **Still not built**: (1) batch Beacon appends via `append_rows()` instead of one `append_row()` call per job, and (2) read the Job ID column once per run into an in-memory set instead of re-reading it per job for the duplicate check. Also worth remembering: `job_log_enabled=false` already exists as an immediate, no-code-change lever to cut this in half by skipping Job Log writes for filtered-out jobs entirely. (The one-time `scripts/rebuild_sheets.py` used for the Beacon/Job Log schema rebuild — see below — does batch its writes via `append_rows()`, but that's a standalone script, not this fix landing in `filter` itself.)

**Why filtering isn't done at ingest time instead** (a real question raised and settled this session): keeping every ingested job in the DB, even ones that fail current filter criteria, isn't wasteful — SQLite doesn't care about row count, and discarding non-matches at ingest time would mean (a) losing the ability to retroactively re-evaluate the backlog if filter criteria change later, since sources don't reliably return the same posting again once it ages out, (b) losing the debugging value of being able to answer "did the pipeline even see this posting" (this exact question came up with Stripe/Glia/Solvd), and (c) breaking cross-source fuzzy dedup, which needs both candidate jobs present to compare. The real cost driver is Sheets writes, not DB storage — addressed above, not by moving filtering earlier.

**No real steady-state ingest rate is known yet.** Every `ingest` run so far has coincided with either a company-list expansion (thousands of jobs, one-time catch-up) or a process being killed/restarted mid-run — there's no clean "just checking for new postings since last time" data point yet. A rough, unmeasured guess once the backlog drains and the company list stops growing: maybe 10-40 genuinely new postings per run combining Adzuna broad discovery and the 99 directly-tracked boards. Worth replacing this guess with real numbers from `jobs.first_seen_at` once the system runs stably for a week or two.

**Pause switch**: both internal schedules (main pipeline, Approval Poller) check `pipeline_paused` at the start of each run — if `true`, log a skipped run to `step_logs` and return immediately, no ingestion, no LLM calls, no Sheet writes. Flip it via the Settings UI or by editing `filter_settings` directly. Useful for going quiet during a slow week without stopping/restarting the process or losing scheduling state.

**Filter engine evaluation order** (cheapest first): keyword match — `role_keyword_include` OR `tech_keyword_include` found in title or description, AND no `title_exclude` term in the title — → seniority → remote/location (incl. `require_us_location`) → posted date within `posted_within_days` → company attributes → **a job that passes all of the above gets a Beacon row immediately** (`add_job_to_beacon` — Job ID, Company, Title, link, location; salary extracted from the description if present; score and visa flag blank) — visa scan and fit scoring update or evict that same row later, they don't gate its creation.

**Seniority and remote_type are inferred, not sourced from any ingestion API.** None of Adzuna/Greenhouse/Lever/Ashby reliably expose a clean seniority or remote-work flag, so the Filter Engine derives both from title/location text the first time a job is evaluated (title markers like "Senior"/"Staff"/"Principal"/"Junior"; location text containing "remote"/"hybrid") and persists the inferred value back to `jobs.seniority`/`jobs.remote_type`. A signal it can't confidently infer is treated as passing, not filtered out — a missing signal isn't evidence of a mismatch, and this app would rather show one extra borderline job than silently drop a real match.

**`require_us_location`** (default `true`) uses a similar fail-open heuristic over free-text location strings (`app/location.py`) — not a geocoder. It recognizes explicit "United States"/"USA" mentions, all 50 US state names/abbreviations (in "City, ST" position), and a curated list of non-US countries/cities/region-codes (APAC, EMEA, etc.) as exclusion signals. A location with no recognizable US or non-US signal (e.g. a bare "HQ" or the ambiguous "NAMER" region code) passes rather than getting excluded. If you'd rather it fail closed on ambiguous cases, that's a one-line policy flip in `app/location.py`, not currently done.

**Real gap found and fixed, larger than expected**: "Dublin" and "Paris" (bare, no country attached) were passing as US locations — two Stripe postings on Beacon that should've been excluded. Auditing the full live backlog for the same class of gap found ~15 more missing cities (Bengaluru, Tokyo, Zurich, Dubai, Mumbai, Chennai, Pune, Gurgaon, Jakarta, Shanghai, Beijing, Oslo, Warsaw, Luxembourg, Tallinn, Reykjavik, Sao Paulo, Bucharest, Herzliya, Phnom Penh, Escazu, Montreal, Zagreb, Split, Doha) plus an entire unrecognized *pattern*: global companies (seen live: Stripe) frequently encode office location as an ISO-3166 alpha-2 country-code prefix -- "AE-Dubai", "IN-Bengaluru", "JP-Tokyo", "CH-Zurich-MSO" -- none of which had ever been caught since no individual city name matched. Added a structural `_COUNTRY_CODE_PREFIX_RE` check for that pattern (covers the whole family, not just the specific cities seen so far), on top of the individual city-name additions. Deliberately did **not** add bare "Manchester"/"Melbourne"/"Vancouver"/"Geneva" as signals, since real US namesakes exist for each (Manchester NH/CT, Melbourne FL, Vancouver WA, Geneva NY/IL) -- used specific compound forms ("Manchester, UK", "AU-Melbourne", "Geneva, GE") instead, verified live that the US namesakes still pass.

**That fix had its own bug, found one session later**: "South Paris, Oxford County" (a real Maine town) was misclassified as Paris, France -- the same real-US-namesake risk already known for Manchester/Melbourne/Geneva also applies to Paris (TX/TN/KY/ME), Dublin (OH/GA/CA), London (OH/KY), Berlin (NH/CT), Warsaw (IN/NY), Madrid (NY/IA), and Lisbon (OH/CT), all of which had been added as unconditional bare signals. Restructured `_segment_is_us` into two tiers: `NON_US_SIGNALS` (no real US namesake, e.g. "Herzliya" -- checked first, so "Herzliya, IL" still correctly resolves as Israel rather than Illinois) and `NON_US_RISKY_SIGNALS` (Paris/Dublin/London/Berlin/Warsaw/Madrid/Lisbon -- checked only *after* a US-confirmation pass: state abbreviation, state name, or a "County"/"Parish" suffix, which is how every real Adzuna US location in this dataset is formatted). A single check order can't get both the Herzliya case and the South Paris case right -- that's why two tiers were needed, not just a reordering.

**Non-US filter-outs don't get a Job Log row at all — per direct request** ("we do not have to log it as filtered out... just ignore it completely"). `FilterResult` carries a `skip_log` flag, set only on this specific branch (not on `location_include`/`remote_type` mismatches, which still log normally); `run_filter_engine` checks it before calling `upsert_job_log_row`. The job is still marked `status='filtered_out'` in the DB as usual (so it's not re-evaluated every run) — only the Job Log Sheet write is skipped, since non-US postings are high-volume noise not worth a row.

**On pulling the last month of postings**: works by default, no special mode — ATS boards return all currently open postings regardless of age; Adzuna is bounded via its `max_days_old` parameter set to `posted_within_days`.

## Job Source Ingestion — Two Modes

**Broad discovery (no company list needed)**: Adzuna's API only accepts query-time parameters for keyword, location, and date — it can't express seniority, salary floor, employee count, or title exclusions, so only three of your filter criteria actually shape the query itself; everything else is enforced after the fetch by the Filter Engine, same as for ATS sources.

**How the query gets built, concretely**:
- **`what`** (keyword): loop over active `filter_keywords` rows across **both** `role_keyword_include` and `tech_keyword_include` categories (see below) — run **one Adzuna query per keyword**, using `what_phrase` rather than `what` for multi-word terms (e.g. "Presales Engineer") so Adzuna matches the exact phrase rather than splitting it into independently-OR'd words, which would over-match. Adzuna's phrase search already checks title *and* description, not just title — confirm exact parameter behavior against Adzuna's current docs during M2, since API details can shift
- **`where`** (location): the active `filter_keywords` rows where `category = 'location_include'`, if any — if empty, omit the `where` param entirely
- **`max_days_old`**: directly from `filter_settings.posted_within_days`
- Full call shape per keyword: `GET https://api.adzuna.com/v1/api/jobs/{country}/search/{page}?what_phrase=<one keyword>&where=<location_include value, if any>&max_days_old=<posted_within_days>`
- **Startup validation**: if `filter_keywords` has zero active rows across both keyword categories combined, Adzuna calls can't run meaningfully — log to the System Log tab and skip Adzuna for that run
- **Volume note**: with 38 keywords × 3 runs/day, that's roughly **114 Adzuna calls/day** — likely still fine on a free tier, but this is enough volume that it's worth actually checking your account's rate limit on Adzuna's developer dashboard during M2 rather than assuming, since the list grew past the point where "probably fine" is good enough. If it turns out to be tight, the cheapest fix is consolidating near-duplicate terms (e.g. "MCP Server" and "MCP Gateway" could arguably merge) rather than cutting coverage
- **Significant limitation confirmed live, not yet acted on**: Adzuna's `/search` API caps the `description` field at **500 characters, server-side** — confirmed by querying the raw API directly, not inferred. Checked against the real data: **all 16,559 Adzuna-sourced jobs in the DB are under 500 chars**, versus a median of ~5,200–7,200 chars (up to 27,667) for Greenhouse/Ashby jobs, which return the full JD. Since Adzuna supplies ~90% of all ingested jobs, this means the Filter Engine's keyword match, the Visa Scanner, the Fit Scorer, and the salary extractor are all working off a short snippet — not the real JD — for the large majority of jobs. Concretely: a keyword that only appears later in the body (a false-negative filter-out), visa/EEO boilerplate (documented elsewhere in this doc as usually sitting near the *end* of a JD), and salary lines are all liable to simply not be present in what's stored. `app.salary_extraction.resolve_posted_salary()` already implements a page-fetch fallback (follows the job's URL, strips HTML, re-runs extraction) with browser-like headers to dodge 403s. Visa-scan got its own equivalent fix first (see below); salary followed later this session, once a direct user report showed the exact same truncation issue hitting a real posting's salary line -- see the Salary Refresh section for the full story, including a real limitation found along the way (some Adzuna redirects can't be fetched at all without executing JavaScript). The Filter Engine's keyword match and fit-scoring still work off the truncated snippet, undecided whether/how to fix -- same tradeoff as always: a network fetch per job slows things down and adds a more fragile failure mode (arbitrary third-party HTML across hundreds of different job boards).

**Salary Refresh (`app/salary_refresh.py`), built per direct report**: "Miller Cooper ... job which description says -- 'A reasonable estimate of the base salary range for this level is $85,000 to $95,000 annualized...' But the range is entirely different on the sheet." Investigated rather than guessed at: the stored description for that specific job was exactly 500 characters, cut off well before that sentence -- the exact same Adzuna truncation root cause already fixed for visa-scan, now confirmed to hit salary extraction too, since `add_job_to_beacon` deliberately only ran the cheap description-only extraction (no page-fetch fallback, to stay fast for every passing job). `resolve_posted_salary()` was the missing piece all along -- built, tested, flagged as a "still open" item in this doc for a while, simply never called from anywhere. New `run_salary_refresh()` calls it for every Beacon job still missing a real salary (`salary_min IS NULL`), tracked via a new `salary_checked_at` timestamp (migration `0006`) so a job is never re-fetched forever, and pushes any found range onto the existing Beacon row via `update_salary_range()`. Runs every main-pipeline cycle, capped at a new `salary_refresh_batch_size` setting (200/run, same "latency cap, not a real quota" reasoning as `link_check_batch_size`) -- 2,593 jobs were eligible when this shipped.

**Real, accepted limitation found while fixing the reported case specifically**: fetching the Miller Cooper job's own posting page directly (with proper browser headers, which already dodge the usual Adzuna 403) returned a generic interstitial page, not the real JD -- its `redirect_url` turned out to be a **client-side JavaScript redirect** to an external board (ZipRecruiter), which `requests` cannot follow since it doesn't execute JS. Confirmed by contrast: a *different* Adzuna job's `redirect_url`, tested earlier this session for an unrelated reason (the link-check investigation), resolved via a normal server-side redirect to an Adzuna-hosted page with the full JD embedded as real text -- both patterns exist, and there's no way to tell which one a given job will hit without trying it. The reported Miller Cooper job had to be corrected by hand (the real $85,000-$95,000 range, already known from the user's own quote, written directly into the DB and pushed to the Sheet) since automated re-fetching genuinely cannot reach its real text. This isn't a wasted fix for everything else, though -- running the new refresh against the first 200 eligible backlog jobs found a real salary for 55 of them (27.5%) via the page-fetch path that *does* work, with sane-looking ranges across a spot-check (e.g. a Sales Engineer role at $86,000-$126,000, a Principal Sales Engineer at $170,000-$281,000). No headless-browser fix planned for the unfetchable minority -- a much bigger investment than any other network call in this pipeline, for a gap the existing Adzuna-estimate fallback already covers reasonably in the meantime.

**Filtering is description-aware, not title-only — this applies everywhere, not just Adzuna**: every keyword match in this system — Adzuna's `what_phrase` (which searches Adzuna's own listing text, title and description both) and the Filter Engine's post-fetch check for ATS-sourced jobs — scans **both `jobs.title` and `jobs.description`**, never title alone. This matters most for the `tech_keyword_include` list, since terms like "Kafka" or "MuleSoft" are far more likely to appear in a job's body text than in its title.

**Keyword matching — two dynamic, independently-editable lists, OR'd together**:
- `filter_keywords.category = 'role_keyword_include'` — job titles you're targeting (e.g. "Solutions Architect," "Presales Engineer")
- `filter_keywords.category = 'tech_keyword_include'` — technology/domain terms (e.g. "API Management," "Event Driven Architecture")
- A job passes this stage if **either list** has at least one match, checked against **both `jobs.title` and `jobs.description`**, case-insensitive, **anchored at the start of a word** (`_contains_any_word` in `app.filter_engine`) — not a raw substring check. This applies uniformly whether the job came from Adzuna (already narrowed at query time) or an ATS source (no query-time filtering, so this check happens entirely in the Filter Engine post-fetch)
- **Real bug found and fixed, two variants**: an ICU nursing posting made it onto Beacon because `tech_keyword_include`'s short keyword "X12" (an EDI transaction standard) substring-matched inside "Weekx12" — Adzuna's scraped text had run the posting's "13 Week" duration and "x12" shift-count together with no space, unrelated to EDI entirely. Fixed by requiring a word boundary immediately *before* a match. Deliberately **not** anchored at the end too, since role/tech keywords are meant to still match their common suffix/plural forms — "Architect" must keep matching inside "Architecture", "Engineer" inside "Engineering", "Integration" inside "Integrations" (a first attempt that anchored both ends broke all of these; checked live against the actual Beacon backlog before shipping the fix — only the one bogus nursing match changed, everything else that used to pass still does). A second, real SSM Health posting ("Respiratory Care Practitioner") still slipped through afterward via "1 to 2 x12-Hour Shifts/Week" — a genuine leading word boundary exists there too (preceded by a space), so the first fix didn't catch it. Fixed by also excluding a match immediately followed by "-hour"/" hour". `title_exclude` below deliberately keeps the plain substring check, since "Intern" is meant to also catch "Internship" titles and "Contract" is meant to also catch "Contractor"
- **Real bug found and fixed: a tracked company's own name as a keyword**: Kong and Boomi are both tracked companies (API-Gateway/iPaaS vendors) whose own name is *also* a `tech_keyword_include` entry -- every posting from them mentions their own name in ordinary boilerplate ("...loyalty to Kong," "Boomi Embedded"), so literally any role at that company (Renewal Account Representative, Sales Development Representative, Product Marketing Manager) trivially "matched" regardless of relevance. Fixed in `evaluate_job` by excluding any keyword that case-insensitively equals the posting's own company name before checking for a match -- the same keyword still counts normally for a *different* company's posting that genuinely requires that tool as a skill
- **A second, unrelated "Kong" collision found later, this time with a place name**: reported as "Executive Chef | Cantonese Cuisine ... chef has to be filtered out." Traced past the reported title to the real cause -- "Kong" was substring-matching "Hong Kong" in the job's *description* ("...Hong Kong-inspired restaurant..."), completely unrelated to the Kong company or the self-name collision above (this restaurant obviously isn't Kong). Not an isolated case once found: **13 completely unrelated jobs were live on Beacon this way** (an MBA leadership program ×3, a bilingual wholesale rep, a bare "Remote" title, an "SAP IBP Architect" posting, etc.), almost none of which had been individually reported. Fixed by generalizing `_contains_any_word`'s existing X12/"-hour" exclusion (previously a single hardcoded case) into a small per-keyword lookup table (`_LOOKAHEAD_EXCLUSIONS`, `_LOOKBEHIND_EXCLUSIONS`) -- "Kong" is now excluded when immediately preceded by "hong "/"hong-", the same way "X12" is excluded when immediately followed by "-hour"/" hour". "Chef" was also added directly to `title_exclude` per the report, as a backstop for any future posting that happens to be genuinely culinary and matches some other keyword. All 13 affected jobs were retroactively evicted by re-running the actual, now-fixed `evaluate_job` against them live (not a hand-rolled substring recheck), confirming none had an independent, legitimate keyword match keeping them valid on other grounds
- Both categories are fully dynamic — add, remove, or edit rows anytime via the Settings UI (`POST /settings/keywords`, `category` = either value) or by editing `filter_keywords` directly. No redeploy needed; the next scheduled run picks up changes automatically
- `title_exclude` remains a separate, title-only safety net applied after the inclusion check -- grown substantially in one session (per direct request, job by job): "Intern," "Contract," "Staffing" (original) → "HVAC," "Industrial Sales Engineer" → "Office Assistant," "Technical Instructor" → "Therapist," "Clinical Supervisor," "Clinician," "LMFT," "Parent Partner," "Assessment Counselor," "Registered Nurse" → "Agent Marketplace," "AI Enablement Specialist," "GTM," "Product Marketing," "Business Development Manager," "Creative Director," "Sales Enablement," "Support Experience," "Developer SDKs," "Sales Strategy and Operations Manager," "Staff Solutions Engineer - Commercial," "Engineering Manager, HADR." "HVAC" and "Industrial Sales Engineer" were added after noticing HVAC/power-generation-equipment sales postings on Beacon: "Technical Sales Engineer"/"Sales Engineer" are genuine `role_keyword_include` matches (the titles really do say that), but the roles themselves are industry-agnostic, so this isn't a keyword-matching bug -- `title_exclude` is exactly the right tool. "Office Assistant" and "Technical Instructor" were added the same way after Workato postings for those (training/admin, not a fit) passed because the company's own "About Us" boilerplate text (present in every one of that company's postings) mentions "iPaaS". The last twelve were added after a batch of Kong/Stripe sales/marketing/ops/support postings turned out to match purely via boilerplate mentions of generic keywords ("Integration," "Messaging," "Solutions Engineer") -- distinct from a real keyword-matching bug (see Job Source Ingestion's Filter Criteria notes below): "Integration"/"Messaging" are simply too-common English words to avoid matching ordinary corporate boilerplate ("refine messaging," "system integration risks," "brand integration"), and tightening them would cost real matches elsewhere, so `title_exclude` on the specific offending titles was the practical choice over redesigning the keywords themselves. **Later, a blanket "Director" and "Manager" were added** (per direct request: "I cannot take managerial or director roles" -- a visa constraint on career level, not a per-title preference) -- these supersede ~12 of the narrower Manager/Director-specific terms above (Business Development Manager, Creative Director, Sales Strategy and Operations Manager, Engineering Manager HADR, and others added since), which were deactivated in the DB as now-redundant rather than left duplicated (seed-filters is add-only, so removing a line from `seed_filters.yaml` never retroactively deactivates it -- that always needs a direct `UPDATE filter_keywords SET is_active = 0`). **Everything else** — `title_exclude`, `seniority`, `remote_type`, `employee_count_min/max`, `founded_after_year`, `company_priority_min`, `industries_include`, salary floor — is applied entirely by the **Filter Engine after ingestion**, identically whether the job came from Adzuna or an ATS source. Filter logic stays in one place rather than half-duplicated into each source's query-building code.

**Targeted tracking (requires a short company list)**: Greenhouse, Lever, Ashby, and SmartRecruiters only expose per-company endpoints — no query parameters to filter by, they return every open posting for that company. All filtering happens post-fetch via the Filter Engine:
- **Greenhouse**: `GET https://boards-api.greenhouse.io/v1/boards/{company}/jobs`
- **Lever**: `GET https://api.lever.co/v0/postings/{company}`
- **Ashby**: public per-company job board API
- **SmartRecruiters** (added per direct request, "our APIs are not effective... add more direct ATS sources"): `GET https://api.smartrecruiters.com/v1/companies/{company}/postings`, paginated 100/page. Structurally unlike the other three -- this list endpoint excludes both the description and a human-facing posting URL, each requiring a separate `GET .../postings/{id}` detail call. Fetching detail unconditionally for every posting on every poll would mean 500-1000+ extra HTTP calls per cycle for a large board (confirmed live: Equinox alone has 666 open postings) -- `app.sources.smartrecruiters.fetch_smartrecruiters_jobs` takes an optional `known_urls` set and only fetches detail for postings not already tracked; an already-known posting only needs its URL reconfirmed (for closed-job detection), not a fresh description. The posting URL itself doesn't need the job's name-derived slug (`https://jobs.smartrecruiters.com/{company}/{id}` resolves 200 without one, confirmed live), so it's built directly from the paginated list without an extra call. `app.ingest.ingest_targeted_company` special-cases this source to compute and pass `known_urls` from the DB before calling the fetcher

Use this selectively for S/A-tier priority companies where you want every posting the moment it's live. Can start empty and grow over time — not a blocker to running the pipeline.

Dedup on `jobs.url`. Closed-job detection (a re-poll finds the job's URL missing from the board's current listing → `status='closed'`). Retry/backoff per source; if a source fails repeatedly, **write an alert row to the Sheet's "System Log" tab** (see Google Sheets Integration below) rather than a Telegram alert — native notification rules cover this same as everything else. HTML stripping before any LLM call. Fuzzy dedup across sources by `company_id` + normalized title + similar location.

**Real, live gap found and fixed this session, built differently from the original plan above** ("write 'Closed - Removed by Employer' to the Sheet's Status column" — that in-place label was never actually built; eviction was): `app.ingest.detect_closed_jobs` correctly flipped `jobs.status` to `'closed'` on a re-poll diff, but nothing ever removed that row from Beacon — confirmed live, **14 jobs already marked `closed` in the DB were still showing on the live sheet**, indefinitely, since nothing ever acted on the flag. User report that surfaced it: "How do we ensure that these jobs are still available? Some links said they are not available anymore." Fixed by wiring the exact same eviction pattern visa-scan and fit-scoring already use (`app.sheets.remove_main_row` + `app.job_log.upsert_job_log_row`, reason `STAGE_CLOSED = "Posting Closed"`) directly into `detect_closed_jobs` — but only when the job was actually on Beacon; one still sitting at `status='new'` just needs the flag flip, since `run_filter_engine`'s own `WHERE status = 'new'` query naturally stops seeing it. All 14 already-affected jobs were backfilled/evicted immediately.

**A second, much bigger gap, previously undocumented anywhere**: this re-poll-and-diff approach only works for Greenhouse/Lever/Ashby, which return their complete current-open set on every poll — it fundamentally cannot apply to Adzuna, whose API is a bounded keyword search with no "list this employer's currently open postings" endpoint at all. Checking the live numbers made clear this isn't a minor edge case: **1,901 of 2,433 jobs currently on Beacon (78%) are Adzuna-sourced**, meaning the overwhelming majority of the list had *zero* closed-job detection of any kind before this session — almost certainly the actual source of the dead links reported. Built a new module, `app/link_check.py` (plus migration `0005_add_link_checked_at.sql` for a per-job "last verified" timestamp), taking a completely different approach: rather than re-polling a board listing, it re-fetches each such job's own posting URL directly and checks whether that specific page now indicates the job is gone.

**Investigated Adzuna's actual real-world behavior live before writing any detection logic, rather than guessing at status codes**: a bare `requests.get()` to an Adzuna `/land/ad/...` URL returns `403 Access Denied` (anti-bot protection) regardless of the job's status — confirmed this isn't a signal of anything. Adding the same realistic browser headers already used for salary extraction's page-fetch fallback (`_BROWSER_HEADERS` in `app/salary_extraction.py`, promoted to a shared `BROWSER_HEADERS` in `app/http_client.py` now that two modules need it) gets past that. With proper headers: a definitely-still-open job (freshly posted, checked live) redirects to `/details/...` and returns **HTTP 200** with no unusual text; a job confirmed to have actually closed redirects to the same kind of URL but returns **HTTP 404** *while still rendering a complete HTML page* (presumably for SEO) containing the exact literal banner text **"Unfortunately, this job is no longer available."** `app.link_check.check_link_dead()` checks for that specific phrase first (the unambiguous signal), falling back to a bare 404/410 status code for non-Adzuna URLs where Adzuna's own wording obviously won't appear. Deliberately does **not** attempt to match a broader set of "this posting is closed"-sounding phrases, since job boards phrase this too many different ways and a loose text heuristic risks the worse failure mode: wrongly evicting a posting that's actually still open. Any network error or timeout is treated as inconclusive — `link_checked_at` is still stamped (so the same job isn't retried again immediately), but nothing is ever evicted on an inconclusive result.

Scoped to jobs currently on Beacon that aren't from a greenhouse/lever/ashby source (so it never duplicates `detect_closed_jobs`'s own coverage), prioritizing whichever jobs have gone longest without a check (`NULL` — never checked at all — first). A new `link_check_batch_size` filter_setting (default 200) caps how many get checked per run — unlike `daily_enrichment_limit`, there's no external API quota to protect here, so this cap exists purely to avoid adding excessive latency to a single pipeline run (each check is one HTTP request with retry, up to ~15s worst case); at 200/run × 3 runs/day, the initial ~1,900-job backlog clears in about 3 days, then settles into a steady-state trickle of new jobs plus periodic re-checks of old ones. Wired into `run_full_pipeline` as a new step, positioned right before the sort+resync step (see Google Sheets Integration below) so that step's row-number resync correctly reflects link-check's evictions too, not just the pipeline's other row changes.

**Third mode — Seed-via-Sheet (built, `app/seed_via_sheet.py`)**: onboarding a targeted-tracking company by typing its name, without knowing anything about Greenhouse/Lever/Ashby ahead of time. Motivation, verbatim: "I keep getting alerts on LinkedIn. I can just put the name of the company on the sheet."

- **Trigger**: a Beacon row with Job ID = `SEED` and the company name typed into Company, everything else left blank. No dedicated input tab — Beacon itself is the input surface.
- **Discovery**: guesses a handful of plausible board slugs from the name (strip common corporate suffixes — "Inc"/"LLC"/"Corp"/etc — then squash spaces together or hyphenate them) and probes Greenhouse, then Lever, then Ashby, then SmartRecruiters with each candidate, cheapest/most-common source first.
- **Verification**: a candidate is only accepted if the probe call succeeds, returns at least one job, AND the guessed slug actually appears in that job's own returned URL — guards against a stale/misleading 200 response for a slug that doesn't really belong to this company (ATS redirects, generic placeholder boards). This was a deliberate design choice after the earlier "Symphony AI" false-positive lesson from manual seed-list curation (see Company Enrichment Data Sources) — cheap insurance against the same kind of mistake happening unsupervised.
- **On a match**: creates the company (or reuses an existing one, matched the same normalized-name way as `app.companies.get_or_create_company`) with `source_type`/`board_url` set and `priority_tier='A'` — same tier as the other manually-curated batches — and writes a one-line outcome into the Title cell, e.g. `"Added via Greenhouse (12 open role(s))"`.
- **On no match**: writes a one-line failure message into Title instead, so you know to add `board_url` by hand if you happen to know it.
- **Cleanup**: a row whose Title is already non-blank (its outcome was already written and seen) gets deleted on the *following* run — it was never a real job posting, just a one-time onboarding request. Processed bottom-to-top within a run so a deletion doesn't shift the row numbers of rows still to be handled below it.
- **Scheduling**: runs automatically as the **first** step of the scheduled pipeline, ahead of `ingest` — a company it onboards gets its postings pulled in during the very same run, not the next one. Also available manually via `python -m app.cli seed-via-sheet`. Always free — no LLM call anywhere in this path.

## Visa Sponsorship Detection Logic

**Important scope note, clarified this session**: this entire step only ever reads the text of the specific job posting itself (title, location, description) — it never checks any real government or third-party record of a company's actual sponsorship history (DOL LCA filings, USCIS H-1B data). "Sponsored" means "this posting's text says something like 'we will sponsor a visa,'" not "this company has a track record of sponsoring." See the DOL/USCIS note under Company Enrichment Data Sources for the (currently unbuilt) alternative.

**Built — a three-tier classification, cheapest first**, to cut Haiku usage down from the ~99.7% of postings it used to handle:

1. **Regex pass** (free): explicit restriction/sponsor-friendly phrases, as today.
2. **Free keyword pre-check**: if regex isn't confident, check whether the description contains *any* of `visa`, `sponsor`, `citizen`, `h1b`, `work permit`, `visa permit`, `work sponsor`, `visa transfer` (case-insensitive substrings) at all (`app.visa_scan.mentions_sponsorship_keywords`). If none are present anywhere, classify immediately as **`No mention`** — no Haiku call, zero cost. This keyword list should keep being checked against real postings as they come in; not treated as permanently final.
3. **Haiku classification** — now only for the smaller remaining bucket where a keyword from the list above *is* present but not in a clean regex-matchable phrase (the genuinely ambiguous cases). The moment a job is determined to need this, its Visa Flag is set to **`Visa Check Pending`** (in the DB and on the Sheet) *before* the Haiku call happens — if the call fails for any reason (including a credit-balance issue), it just stays at `Visa Check Pending` and gets retried automatically on the next `visa-scan` run, rather than silently staying blank or, as a real bug this fixed along the way, aborting every job after it in that same run's batch. Once Haiku responds: `{"visa_flag": "restricted"|"sponsors"|"unclear", "snippet": "..."}`.

**Visa Flag values on the Sheet** (relabeled from the internal `restricted`/`sponsors`/`unclear`/`no_mention`/`pending` at write time only, via `app.sheets.display_visa_flag`): `Sponsored`, `No sponsor`, `No mention`, `Visa Check Pending`, `Unclear` (Haiku's own "read it, still can't tell" verdict — distinct from `No mention`, which never even reached Haiku).

Store result + snippet in `jobs.visa_flag`/`jobs.visa_snippet` for audit regardless of downstream filtering. **Runs as part of the main pipeline (3x/day), after the job already has a Beacon row.** Updates that row's Visa Flag cell in place. If `restricted` and `require_visa_sponsorship` is on, the row is evicted from Beacon and moved to the Job Log ("Filtered") sheet with reason `Visa Restricted` instead — same eviction pattern as sub-threshold fit scores (see Fit Scoring Logic)

**Historically the regex pass was confident on roughly 1 in 300 real postings**, with Haiku carrying almost all classification load as a result — the motivation for the free pre-check above. In practice, a large fraction of postings that mention nothing about sponsorship now resolve to `No mention` for free instead, and only postings that actually reference sponsorship in some form but aren't cleanly parseable by regex cost anything.

**Full-page fetch wired in before all three tiers above**, fixing a real, root-caused gap (user report: "Lot of descriptions have -- 'At this time, we typically do not offer visa sponsorship...'" that wasn't getting flagged). Investigated rather than guessed at: queried the live DB and found the *median* stored `description` length across all 2,380 jobs then on Beacon was exactly 500 characters -- Adzuna's `/search` API truncates `description` server-side at exactly that length, affecting ~90% of ingested jobs (everything from Adzuna broad discovery, not the direct Greenhouse/Lever/Ashby board scrapes). The 1,493 jobs classified `no_mention` at the time also had a median length of exactly 500 -- meaning the keyword pre-check correctly found nothing in text that never contained "sponsor," "visa," etc. at all, because a sponsorship disclaimer (almost always EEO/legal boilerplate placed near the *end* of a real posting) had already been cut off before the description was ever stored. The three-tier classification logic itself was never wrong -- it just never saw the deciding sentence for the majority of jobs.

Fixed by calling `app.salary_extraction.fetch_job_page_text()` -- an existing, tested helper originally built for `resolve_posted_salary()`'s salary-extraction fallback, but never actually wired into anything before now (a "still open" item carried in this doc for a while) -- for every job in `run_visa_scan()`, before classification. On success, the full page text replaces the stored (possibly truncated) description for regex/keyword-check/Haiku purposes only (`jobs.description` itself is untouched, since other steps like the Filter Engine and fit-scoring have their own separate, not-yet-addressed relationship with this same truncation problem). On any fetch failure (dead link, network error, non-200 status) it falls back silently to the stored description, logged as `page_fetch_failed` via `app.observability.log_step` -- one bad URL never blocks the rest of a scan run, same "one bad job must not kill the batch" pattern used everywhere else in this pipeline. Deliberately chosen as unconditional-for-every-job rather than a narrower "only fetch when truncation looks likely" heuristic (e.g. description length == 500), since a length heuristic risks false negatives on genuinely-short real postings and false positives on ones that legitimately end around 500 characters -- the extra HTTP request per job (slower visa-scan runs, another network dependency that can fail) was accepted as the simpler, more complete tradeoff. 176 jobs were queued for visa-scan at the time this shipped, so the first run afterward took noticeably longer than the steady-state 3x/day incremental volume.

**Real bugs/gaps found and fixed via the golden test set, worth knowing about if the prompt or truncation limit ever gets touched again:**
1. Visa/EEO/sponsorship boilerplate is frequently near the **end** of a JD, not the start. An earlier 4,000-char prompt truncation cut off the relevant sentence entirely on a real posting, causing a false "unclear". Truncation is now 12,000 chars — Haiku is cheap enough that this isn't a meaningful cost concern.
2. The Haiku prompt must include the **job's own title and location**, not just the description. A real posting said "we can sponsor visas to Germany; for any other country, you need existing right to work" — reused as boilerplate across that company's global postings. Without location context, Haiku read "we sponsor visas" as a positive signal on a US-located posting where it actually meant the opposite. Both `haiku_classify()` and the golden test set's flagship case (`real-151-germany-only-restricted`) cover this.
3. The `without ... sponsorship` regex pattern only allowed the word "visa" between "the need for" and "sponsorship" — missing the extremely common real phrasing "without the need for **employer** sponsorship, now or at any time in the future" (reported live from an actual posting), which fell through to Haiku instead of being caught for free by the fast regex path. Fixed by adding "employer" as an alternative in that slot; covered by golden case `real-employer-sponsorship-future-restricted`. Spot-checked two more real examples against the classifier at the same time ("— no visa sponsorship available", "unable to sponsor H-1B, F-1 OPT, and STEM OPT extension") — both already classified correctly, no fix needed.
4. Two more real, reportedly very common phrasings weren't caught by any existing pattern, since all of them expect the negation word directly before "sponsor(ship)" (e.g. "not sponsor"): "Visa sponsorship **is not available** for this position" (negation comes *after* the noun) and "...**does not now, or in the future, require** visa sponsorship..." (standard EEO/legal boilerplate, a different sentence construction from the existing "without ... sponsorship, now or at any time" pattern). Two new dedicated `RESTRICTION_PATTERNS` entries added; covered by golden cases `real-sponsorship-is-not-available-restricted` and `real-does-not-now-or-future-require-sponsorship-restricted`.
5. **Two more real phrasings, found the following session**: "This role is **not open to** (visa) sponsorship" -- doesn't fit the existing "not sponsor(ship)" pattern, which only allows "currently"/"provide"/"offer" in that slot, not "open to" -- and "...**without requiring a visa transfer or** visa sponsorship" -- pairs "visa transfer" (an H-1B moved between employers, a distinct restriction from sponsorship) with "sponsorship" in the same "without ... or ..." clause, which the existing "without ... sponsorship" pattern didn't have room for. Each got its own `RESTRICTION_PATTERNS` entry and a permanent regression test (`test_regex_classify_detects_not_open_to_sponsorship`, `test_regex_classify_detects_without_visa_transfer_or_sponsorship`). Two other reported phrasings that session ("government security clearance... US Citizen", "unable to sponsor H-1B, F-1 OPT, and STEM OPT") were checked against the classifier and found to already work correctly -- no fix needed for those two.

**Golden test set is now 11 cases** (up from 8) after these fixes.

## Fit Scoring Logic

**"Score Request" (v1) has been fully replaced by a richer column, "My Decision" — built.**

**My Decision** is a full state machine instead of a single Yes/blank flag:
- `New` — python sets this by default when a job lands on Beacon
- `Go Score` — you set this to request scoring (replaces the old "Score Request: Yes")
- `AI Score Pending` — python sets this the moment a `fit-score` run claims the job for processing, *before* the actual API call (`update_my_decision`). If the call fails (parse error, budget runs out mid-run), it **stays** at `AI Score Pending` rather than reverting — the next run retries it automatically
- `AI Scored` — python sets this once the score is actually written (`update_score` sets this internally too)
- `Manual Scored` — you set this yourself, independent of the AI path entirely (you scored it yourself, no API call happened)
- `Reject` — you set this to evict the row to the Job Log; handled by `app.fit_scoring._process_rejections()`, which runs at the start of every fit-scoring pass regardless of whether anything else is also flagged for scoring

This `My Decision` column is **separate from the existing `Decision` column** (Pending/Approve/Deny), which is unchanged and still exists purely as the approval gate for a future resume-handoff (M7) — the two track genuinely different things: My Decision is about the scoring/review workflow, Decision is about "should I apply." `get_scoreable_job_ids()`/`get_rejected_job_ids()` bulk-read the flagged sets from the Sheet; `run_fit_scoring()` returns immediately, touching neither the DB nor the API, if nothing is flagged or rejected.

**Scheduling — built**: fit-scoring moved off its overnight-only 2am slot entirely. It stays its own independent scheduled job (own `workflow_run`, not literally merged into the main pipeline's code), but its `CronTrigger` now fires at the same three times as the main pipeline (8am/1pm/6pm). The original reasoning for isolating it overnight — avoiding competition with same-day filter/visa-scan spend for the shared `daily_token_budget` — mattered when scoring ran automatically against the whole backlog; now that volume is bounded by how many jobs you've manually flagged, that isolation is no longer worth the up-to-24-hour delay between flagging a job and it getting scored.

- Input: JD + base (or best-matching registered tailored) resume
- Claude API call requesting `{"score": 0-100, "matched_skills": [...], "gaps": [...], "summary": "..."}`
- Store in `fit_scores`
- **The job already has a Beacon row by this point** (added the moment it passed the Filter Engine — see Job Source Ingestion / Filter Criteria). Fit-scoring never appends a new row; it only updates or evicts the existing one:
  - `score > fit_score_threshold`: update that row's Initial Fit Score cell in place (My Decision → `AI Scored`), leave it on Beacon. Score cell gets a light-blue conditional-format highlight so scored rows are visually distinct from ones still awaiting a score
  - `score <= fit_score_threshold`: mark My Decision `AI Scored` first, then **evict** — remove the row from Beacon (`ws.delete_rows`), clear `jobs.sheet_row_number`, and write/update a row on the Job Log ("Filtered") sheet instead with reason `Scored Below Threshold` and the score — see Job Log section below
- Throttled by `daily_token_budget`/`monthly_token_budget` same as before, though in practice volume is now naturally small since it's manually gated
- Every DB write in this loop is `commit()`-ed *before* the Beacon/Job Log Sheets call that follows it — see the SQLite `busy_timeout` note in Tech Stack for why this matters

**Cost guardrail**: check cumulative spend against `daily_token_budget`/`monthly_token_budget` before each LLM call; if exceeded, pause remaining LLM-dependent steps for the run and **write an alert row to the Sheet's System Log tab** — native notification handles delivery. The tracker must read prior spend from `workflow_runs.estimated_cost_usd` across separate scheduled runs, not just the current run's in-memory total — an earlier implementation computed cost per-call but never persisted the run total, silently defeating the daily/monthly ceiling across runs. Fixed; covered by `tests/test_budget.py`.

**Known limitation: Sonnet 5 has no `temperature` parameter** (it's in the model family where sampling params were removed in favor of prompting/effort — sending one returns a 400, unlike Haiku 4.5 which still accepts it). There is no determinism knob for fit scoring. Real observed variance: the same JD + resume scored 47 on one call and 72 on the next with no code changes. A prompt instruction ("enumerate JD requirements and check each against specific resume evidence, rather than forming a general impression") helps somewhat but doesn't eliminate this. A fit score reflects a single sample, not a stable measurement — worth keeping in mind for borderline (near-`fit_score_threshold`) jobs.

**Golden test set**: `tests/golden_set/cases.json` — 9 cases (7 pulled from real ingested/reported postings with hand-verified expected outcomes, 2 constructed to exercise paths no real posting in the dataset happened to hit). Regex-confident cases run in the normal fast suite; cases needing the real Haiku API are skipped by default (`RUN_GOLDEN_LIVE=1` to opt in) so `pytest` doesn't cost money on every run.

## Approval via Google Sheets

**Decision column setup**: header background green, dropdown data validation (`Pending`/`Approve`/`Deny`), defaults to `Pending` on row creation. Adjacent `Rejection Reason` column — fill in directly when setting Decision to `Deny`.

**Not to be confused with "My Decision"** (built — see Fit Scoring Logic and the Beacon columns table): that's a separate column tracking the scoring/review workflow (`New`/`Go Score`/`AI Score Pending`/`AI Scored`/`Manual Scored`/`Reject`), independent of this one. Everything in this section is about the unchanged `Decision` column and the Approval Poller that acts on it.

**Approval Poller** (internal schedule, same process as ingestion, default every 30 min; logs to `workflow_runs` with `run_type='approval_poll'`; **built — M6**, `app/approval.py` + `app.pipeline.run_scheduled_approval_poll`; manual run via `python -m app.cli approval-poll`):
1. Query `jobs` where `sheet_row_number IS NOT NULL` and `decision_processed_at IS NULL` — **not `status = 'notified'`**, which nothing sets anymore since the Beacon-immediate-add redesign (a job's `status` tracks pipeline progress, not Sheet presence; `sheet_row_number` is the live "is this job on Beacon right now" signal — see Company Enrichment Data Sources for the same scoping change applied there first)
2. For each, read the Sheet's Decision/Rejection Reason/Reminder Sent At cells in one call (`app.sheets.read_decision_row`) at `sheet_row_number`
3. `Pending` + within reminder window → no action
4. `Pending` + past `approval_reminder_hours` (default 48) **and "Reminder Sent At" is still empty** → update that row's "Reminder Sent At" column with the current timestamp — this edit triggers Google's native notification email. **Fires exactly once per job, not once per poll cycle**: without this guard, a job left "Pending" for a week would get a reminder every single 30-minute poll (48+ emails), since the reminder condition would keep re-triggering. Checking that "Reminder Sent At" is still empty before sending means you get one gentle nudge, then the app leaves it alone until you act — you can still see it's overdue just by looking at the Sheet whenever you check. **Real gap fixed along the way**: the reminder-window baseline is `jobs.notified_at`, which was defined in the schema from the start but never actually set anywhere in the code until M6 — `add_job_to_beacon()` now sets it (backfilling via `first_seen_at` for jobs added before this fix, so old Beacon rows still get correctly-timed reminders instead of never qualifying)
5. `Deny` → read Rejection Reason cell (defaults to "no reason given" if blank), `jobs.status='rejected'`, `decision_processed_at=now`
6. `Approve` → **records the decision only today** (`jobs.status='approved'`, `decision_processed_at=now`, logged to `step_logs`) — the actual Claude Desktop handoff (clipboard, deep link, Handoff Prompt column) is still M7 and not built; M6 was scoped to the poll/reminder/decision-recording mechanics only
7. Log every check to `step_logs`, including "still pending, no action"

**Failure handling — Sheets API unreachable**: if a poll cycle can't reach the Sheets API (auth error, network error, expired service account credentials), log it as a failed `workflow_runs` row (`run_type='approval_poll'`, `status='failed'`). After `approval_poller_max_consecutive_failures` (default 3) consecutive failures, write a prominent `CRITICAL` entry to `step_logs` — checked via `count_consecutive_failures()` walking recent `approval_poll` rows in `workflow_runs`. **Note: `last-run` CLI (referenced here and elsewhere in this doc as the way to surface this) was never actually built** — for now, check `workflow_runs`/`step_logs` directly in `job_search.db`, or `scheduler.log`.

**Idempotency**: `decision_processed_at` is the guard — a job is never reprocessed once acted on, even if the Decision cell still shows `Approve`/`Deny` later.

## Google Sheets Integration (Tracking, Approval, AND Notification)

**Why the app writes to Sheets directly**: no live Claude session runs during unattended execution — the Python process calls the Sheets API via a **service account**, credentials in `.env` as `GOOGLE_SHEETS_CREDENTIALS_PATH`.

**Notification is entirely native to Sheets — no app-side notification logic beyond writing data**: enable **Tools → Notification rules → "Any changes are made" → "Email — right away"** once, yourself, in the Sheets UI. Because the app writes via a service account (a distinct Google identity from you), every automated write — a new match, a reminder timestamp, a handoff confirmation, a system log entry — counts as "someone else's" edit and triggers an email. Your own manual edits (setting Decision, updating Status) do **not** trigger notifications back to yourself, so this doesn't create noise from your own actions. **Verify both directions explicitly during M5** rather than assuming — this is a real behavioral assumption, not a documented guarantee for this exact setup. Status: app-write → email confirmed live; self-edit → no-email still needs a manual check.

**Known limitation, accepted as a tradeoff for simplicity**: Google's native "Any changes are made" email is typically generic — something like *"A change was made to [Sheet name]"* — not a formatted message with the job title, company, and score. You'll need to click through to the Sheet to see what actually happened, every time. This is a real regression from a richer message-based notification, deliberately traded away to avoid building and maintaining a bot. If this proves too vague in practice after using it for a while, the cheapest upgrade path is a Google Apps Script `onChange` trigger that composes a richer email from the row's contents — not something to build preemptively, only if the generic version turns out to actually bother you.

**Beacon is auto-sorted alphabetically by Company at the end of every main pipeline run** (per direct request, "Sort alphabetically on sheets at the end of the job"). `app.sheets.sort_and_resync_main_sheet()` sorts the data rows (header protected via an explicit range, since this sheet has 0 frozen rows so gspread's default "sort everything except frozen rows" behavior can't be relied on), then re-reads the Job ID column and corrects every job's `sheet_row_number` in the DB. That resync step turned out to matter for more than just sorting: it was built after discovering that `app.approval`'s `read_decision_row`/`update_reminder_sent_at` are the *only* Sheets-writing functions in this codebase that read/write `job["sheet_row_number"]` directly instead of re-locating the row fresh via `find_existing_row` first (every other one -- `update_visa_flag`, `update_score`, `update_company_columns`, `remove_main_row` -- already does). That means any eviction at all (visa-restricted, sub-threshold fit score, manual Reject -- all pre-existing, nothing to do with sorting) was already silently drifting the Approval Poller's target row for everything below it out of sync with the DB, a real bug that predates this feature. Fixed by pulling the resync into its own `resync_sheet_row_numbers()` function and calling it a second time (without sorting) at the end of `run_scheduled_fit_scoring`, since fit-scoring runs on its own separate schedule and evicts rows via the same `remove_main_row` mechanism. A one-time manual sort + row-number remap (2,173 rows, spot-checked against the live sheet) was performed before this was automated, to establish it was safe.

**A real, related bug found and fixed while adding the "Posted Today"/"Posted This Week" highlighting above**: every one of the 5 conditional-format rules that existed before this (Approve/Deny row color, Score highlight, Industry deprioritize, Salary estimate pink) had a range that omitted `endRowIndex` -- which the Sheets API does NOT treat as "grow forever with the sheet," it resolves to the sheet's row count *at the moment the rule was created* and freezes there. Confirmed live: all 5 were frozen at `endRowIndex=2551`, the exact size the sheet was when `ensure_main_sheet_headers` first ran -- meaning every row appended past row 2551 had already been silently getting **zero** conditional formatting (no Approve/Deny color, no Score highlight, no Industry grey, no Salary pink) for some time before this was caught. The exact same bug existed for `VALIDATION_ROW_COUNT` (the Decision/My Decision/Application Status dropdown ranges), still set to 1000 despite live data already well past that. Fixed two ways: (1) resized the actual sheet grid via `ws.resize()`, since the Sheets API also silently clamps any GridRange's `endRowIndex` to the sheet's real dimensions -- widening the range alone wasn't enough without the grid itself being big enough to hold it; (2) every conditional-format range and validation range now specifies an explicit `CONDITIONAL_FORMAT_ROW_HEADROOM`/`VALIDATION_ROW_COUNT` instead of an open-ended one. Applied live by deleting and recreating all 7 conditional-format rules (5 existing + the 2 new date ones) and re-issuing the 3 validation calls -- not just changing the constant in code, which would only take effect on a future full sheet rebuild.

**Immediate regression, took two live iterations to actually fix (user report: "Google Sheets App is crashing")**: the fix above first shipped with a 50,000-row headroom -- picked as "generously large" without weighing the actual cost of that choice. The Sheets *client* (not this pipeline -- every API call had succeeded normally, nothing showed up in any log here) has to evaluate all 7 conditional-format rules for every row in range, and several of those rules are `CUSTOM_FORMULA` (real per-row formula evaluation, not a cheap static check) -- doing that across 50,000 rows × 20 columns turned out to be enough client-side rendering load to crash the app outright. A follow-up question narrowed it down further: the crash was specifically the **mobile app**, and specifically triggered by **scrolling** -- consistent with progressive client-side re-evaluation as more rows come into view, not a one-time load spike on open. First attempt at a fix reverted to 8,000 rows -- **still crashed on the same test**, which was the real signal that this needed to come down much closer to actual data size (~2,559 rows at the time), not just "smaller than 50,000."

**Built a self-maintaining mechanism instead of picking another fixed number** (`app.sheets.ensure_beacon_capacity()`), since keeping the range close to real data means it now needs periodic small bumps as the sheet grows, rather than one large buffer set once. Runs every main-pipeline cycle via `sort_and_resync_main_sheet` -- cheap in the common case (one column read to check current data extent against the grid size); only does the heavier delete-and-rebuild-7-rules work when data comes within 300 rows of the current limit, growing by a fixed 500-row increment each time. This is deliberately the opposite instinct from the first fix: instead of "make it generously large so I never have to think about this again," the design accepts frequent small maintenance in exchange for never again being either stale (silently drops formatting for new rows) or oversized (crashes the client) for long. The rule-building logic itself was extracted out of `ensure_main_sheet_headers` into a shared `_apply_row_ranged_formatting()` function so the one-time initial setup and the ongoing capacity-growth path can never drift out of sync with each other.

**A real, separate bug found while building the self-maintaining mechanism**: `_salary_estimate_highlight_rule_request()` -- unlike every other rule-builder function in this file -- never accepted a `row_headroom` parameter at all; it always read the module-level `CONDITIONAL_FORMAT_ROW_HEADROOM` constant directly, completely ignoring whatever range its caller actually intended. This went unnoticed through the earlier 50,000→8,000 revert (both were global constant changes, so the bug had no visible effect), but surfaced immediately once capacity became dynamic: after a live capacity fix, 6 of the 7 rules showed the new, correct `endRowIndex` and the Salary-estimate-pink rule alone was still stuck at a stale value from a previous run. Fixed by giving it the same `row_headroom` parameter every other rule builder already had.

**Still crashing after all of that -- because the row range was never actually the root cause (user report: "Still crashing after scrolling").** At this point the range had been cut from 50,000 all the way down to ~3,359 (barely above real data size) across two separate attempts, with zero change in outcome -- a strong signal that row count wasn't the real variable being tested. Ran one clean, conclusive diagnostic instead of guessing a third row-count value: deleted all 7 conditional-format rules from the live sheet entirely, nothing else changed. **User confirmed: no crash.** This pinned the cause precisely on the rules themselves, not the range -- and specifically, given the original 5 rules (Approve/Deny/Score/Industry/Salary) had been running without incident for a long time before this session, the 2 new date-parsing `CUSTOM_FORMULA` rules were the actual culprit. Their formulas were genuinely more expensive than the pre-existing ones: nested `MID`/`LEFT`/`VALUE`/`DATE` calls wrapped in `IFERROR`/`AND`, evaluated fresh by the client for every visible row on every scroll -- exactly the kind of live, per-row computation a mobile Sheets client struggles with at any meaningful row count, independent of how small that count is made.

**Redesigned the whole feature around a static, Python-computed color instead of a live Sheets formula.** New `app.sheets._date_highlight_color(posted_at, today_central)` computes the target color directly from `jobs.posted_at` -- a real datetime already sitting in the DB -- rather than re-parsing the *rendered* Sheet text (`"mmddyyyy HH:MM"`) back into a date via formula, which is what made the old approach expensive in the first place. New `app.sheets.refresh_date_highlights(conn, ws)` re-evaluates every currently-on-Beacon job's color each main-pipeline cycle (wired into `sort_and_resync_main_sheet`, positioned after the row-number resync since it depends on `sheet_row_number` being current) and writes the result as a plain static `repeatCell` background color -- something any Sheets client renders at essentially zero cost, the same as any other manually-colored cell, since there's no formula to evaluate at all. This needs a *full* re-evaluation every single run, not just for newly-added jobs: a job that was "posted today" yesterday needs to fade to "posted this week" green today, and a job from last week's "this week" green needs to clear back to no color once a new week starts -- neither of those transitions has any other trigger, since nothing else in the pipeline ever revisits an existing job's color. The two formula-based rule-builder functions (`_posted_today_highlight_rule_request`/`_posted_this_week_highlight_rule_request`) were deleted outright rather than left dormant, and the live sheet is back to exactly the 5 conditional-format rules that were always known to be stable.

**"Cloud Platforms" column built** (per direct request: detect whether a posting mentions AWS/GCP/Azure). `app.cloud_platforms.resolve_cloud_platforms()` regex-matches title+description for each of the three, computed once at ingest time and appended to the end of both column lists (append-only, per the Location State column-move lesson above -- never insert mid-list on a live sheet). Comparing AWS-vs-GCP-vs-Azure mention counts surfaced that most jobs had no cloud label at all, which traced to the same Adzuna 500-char description truncation already fixed for visa-scan and salary (confirmed: 87.8% of Adzuna-sourced Beacon jobs truncated to exactly 500 chars, vs. 0% for Greenhouse/Lever/Ashby-sourced ones). `app/cloud_platforms_refresh.py` (migration `0009_add_cloud_platforms_checked_at.sql`) re-checks via the full posting page for any job whose stored description is exactly 500 chars, mirroring `app.salary_refresh`'s design exactly, including the same "some Adzuna redirects are unfetchable client-side JS" limitation. New `cloud_platforms_refresh_batch_size` setting (200/run).

**The biggest bug found this session: unbounded `ws.append_row()` calls were silently ballooning the Beacon sheet's column count, and this turned out to be the actual cause of the recurring 10M-cell workbook-limit errors that had silently killed the `filter` pipeline step for 3+ days straight (2026-07-11 through 2026-07-14).** Nothing in any log surfaced this as a Sheets bug at first — the `filter` step just logged a `[400] This action would increase the number of cells in the workbook above the limit of 10000000 cells` warning and moved on each run, so a growing backlog of jobs simply never got filtered/added while this went unnoticed. Investigated methodically once found, ruling out theories in order rather than guessing: not a stale scheduler process with old code loaded in memory (reproduced with the scheduler fully stopped), not a concurrency race between two writers (reproduced with exactly one script touching the sheet), not any explicit `resize`/`insert_cols` call anywhere in the codebase (grepped for every one — the only call that exists touches row count only). Isolated the real trigger via a controlled single-job trace: calling `add_job_to_beacon` for one real job with an explicit `table_range` bound held the sheet's column count perfectly flat across 8 consecutive real appends; the identical appends *without* that bound grew column count on nearly every single call (roughly 1.5-2% per call, consistent with Google's own grid-auto-expand heuristic). Root cause: `gspread`'s `append_row`/`append_rows`, when given no `table_range`, ask the Sheets API to auto-detect "the table" against the whole, unbounded sheet name — and the API's own grid-auto-expand behavior silently grows `gridProperties.columnCount` as a side effect of that. **Fixed** by adding `table_range=f"A1:{last_col}1"` to the only two places in the codebase that ever append a row: `add_job_to_beacon` (`app/sheets.py`) and `upsert_job_log_row` (`app/job_log.py`). Verified live before and after: with the bound, repeated real appends never moved column count; without it, column count ballooned every time, twice reproduced (once to ~2,863 columns, once to ~1,980). All 396 tests pass (6 new). Backlog-clearing after the fix is ongoing, deliberately throttled (small batches, paced) to avoid saturating the Sheets API while the live sheet is in active use — see RUNBOOK.md for current status.

## Job Log ("Filtered" Sheet) — Where Excluded Jobs Go

**Redesigned mid-build from the original "logs everything" concept below.** Originally this sheet logged every job at every stage, including ones that passed (a `Passed Filter - Awaiting Score` row for every job, in addition to the main Sheet). Once Beacon started getting a row immediately at filter-pass time (see Fit Scoring Logic / Filter Criteria), logging *passing* jobs here too became redundant — a job that passes now has exactly one home (Beacon) until/unless it's excluded, at which point it moves here instead. **The Job Log's role narrowed to: where jobs go when they don't (or stop) qualifying for Beacon.**

**One row per job, ever — upsert, not append.** This was also a design change: originally an append-only audit trail (a job could accumulate several rows over its lifecycle). Now each job gets found-or-created by Job ID and that same row is updated in place if its exclusion reason changes — though in practice a job only lands here once, since exclusion is terminal (filtered-out jobs never re-enter the pipeline; evicted-from-Beacon jobs don't get re-added).

**Why this needs a *separate* spreadsheet, not a new tab on Beacon**: Beacon relies on Google's native "Any changes are made" notification firing on every automated write — that's the entire notification mechanism. If every excluded job (potentially hundreds during a backlog catch-up) also wrote to that same spreadsheet, it would flood you with notifications for jobs you were never going to see anyway. So this is a **second Google Sheet, same service account, notifications deliberately left off** — a queryable list you check when curious, not something that pushes anything to you.

**Job Log Sheet columns — redesign built and deployed: mirrors Beacon's new column set almost exactly**, rather than staying as lean as it was before. Rationale: the separate-spreadsheet architecture exists purely for *notification* isolation (Beacon's native "any change" email rule would flood the inbox if every excluded job wrote to the same sheet) — that's independent of column layout, so there's no reason Job Log can't carry the same rich context as Beacon. The live rebuild (same `scripts/rebuild_sheets.py` run) repopulated this sheet from the **database** (767 excluded jobs total) rather than just the 182 rows already on the old Job Log sheet — `job_log_enabled` had been off for a stretch, so the sheet alone was missing most of the real exclusion history; visa-restricted reasons were reconstructed from `jobs.visa_snippet` to match the live format exactly.

## What NOT to do
- Do not auto-submit applications under any circumstance
- Do not build any email-sending, Telegram, or other messaging integration — notification is Sheets-native only (Tools → Notification rules). Approval Poller failures are logged to step_logs and surfaced via the last-run CLI, nothing more
- Do not treat any Sheet column as a control input except Decision
- **Do not enable Google's Notification rules on the Job Log ("Filtered") Sheet** — it's high-volume during backlog catch-ups (every filtered-out and evicted job), and turning on notifications there would flood the inbox and defeat the whole point of keeping it separate from Beacon
- Do not reprocess a job whose decision has already been acted on — check `decision_processed_at`
- Do not call the Claude API for visa classification when regex already gives a confident match, or (per the pre-check redesign) when none of the sponsorship-related keywords appear in the text at all
- Do not build `applications`/`interview_rounds`/`cover_letters`/`resumes` as relational DB tables
- Do not hardcode filter criteria — everything in `filter_settings`/`filter_keywords`
- **Do not run fit-scoring against a job unless the user has explicitly flagged it** (`My Decision = "Go Score"`, or `"AI Score Pending"` when retrying after a prior failed attempt) — never auto-score the whole backlog, regardless of budget headroom
- **Company enrichment must never cost money, full stop** — free sources only (FMP, StartupHub.ai). If they don't have a field, leave it blank; do not fall back to an LLM call to fill the gap, even if budget is available
